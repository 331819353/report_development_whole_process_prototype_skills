# Template Routing And Implementation Gates

Detailed template routing and implementation rules moved out of `SKILL.md`. Load this before selecting, materializing, configuring, editing, or accepting bundled report templates.

Default routing:

- Default to a selected bundled `frameworkTemplateId` for runnable report prototypes.
- Default to the configurable template route: materialize the selected bundled template project, then configure it through the operation chain. Do not start from an independent scaffold inside this workflow; unsupported shell/layout requirements become blockers or template backlog items.
- After materializing the selected bundled template project and before editing source, run `npm run ledger:init` to generate baseline `__change_logs__` sidecars. Before handoff, run `npm run ledger:check`; this project-level check complements the per-file `ledger:code --stage before/after` flow for edited files.
- In the report development flow, do not choose an alternate shell path. Custom/free shell, exact shell restoration, existing shell preservation, custom page layout, custom block layout, and duplicate filter/nav/toolbar surfaces are blockers or out-of-scope exceptions, not implementation routes.
- If the user or PRD provides HTML, Markdown/MD, copied source, or a loose sample/reference, use it as requirement, hierarchy, density, and tone evidence and still choose the closest bundled template. Do not let PRD wording or source format override the template-only report development constraint.
- After data, filters, widgets, generated conclusion rules, and interactions are configured in the prototype, produce `docs/prototype-data-summary.md` as the backend-facing data handoff artifact. Use `prototype-data-summary-contract.md`; the document is required before technical solution, backend/data-service design, frontend integration, testing, or final handoff.
- Before implementation consumes a PRD, create or validate `templateAssetUnderstandingMap` from `template-asset-construction-contract.md`, then create or validate a Template Build Packet from `template-build-packet-contract.md`. The packet must normalize the selected template, shell, pages, asset evidence, `layoutSectionMap`, `layoutRows`, readable `layoutCoordinateMap`, blocks, standard areas, component slots, data/API, filters/actions, interactions, conclusion rules, self-development exceptions, target files, and validation commands into fixed fields. Weak/literal models must implement from this packet row by row instead of searching across PRD prose.
- The only allowed self-developed surfaces are interaction behavior and registered component examples. Record them in `selfDevelopmentExceptionMap` as `interactionBehavior` or `componentExample`.
- Template routing must receive or produce a source-material requirement matrix before implementation when source artifacts are provided. Source artifact format and PRD wording are evidence only; they do not create an alternate implementation route inside this workflow.
- If source material defines block title chrome, component-container background, or "先选分块样式再填内容" behavior, route it through `report-visual-layout-design` `references/block-chrome-style-patterns.md`. Carry `blockChromePattern` into widget/composite config before filling body content; do not copy the raw HTML wrapper as the template implementation.
- Use the standard report-development terms when mapping template assets: 框架模板 selects the shell, 页面布局配置 owns `layoutSectionMap` and `layoutRows`, 分块配置 owns block size/slots plus only the standard block areas `1-1 titleArea` 标题区, `1-2 pillArea` 胶囊按钮区, `3 componentArea` 组件区, and `4 summaryArea` 说明区. Every 分块配置 must be represented by `createBlockAreaConfig` config and rendered by `BaseLayoutSpan.vue`; `componentRegionPattern` may appear only as a derived slot-geometry descriptor. 组件示例 fills only the implemented component's internal content inside `3 componentArea`, must resolve to a standalone Vue file that can be copied or mounted, and must pass visual-type size compatibility for the slot. Component slot fills must not carry filters, controls, parent-block title pills, descriptions, or summary/explanation copy. Component-owned internal props such as `unit` and `auxMetrics` remain valid when the selected registered example renders them.
- Before selecting a block area config, analyze the user's analytical need and decide whether the block should be SingleSlot or MultiSlot. Use SingleSlot for one dominant conclusion card/component. Use MultiSlot only for parallel evidence, comparison, conclusion-card-plus-driver, or tightly related component groups; in MultiSlot templates, the first component position should carry the conclusion card or primary conclusion component when componentized conclusions are needed. Text-only/narrative conclusions stay in `4 summaryArea` only when the block has no conclusion card/component; if a conclusion card/component exists, `4 summaryArea` must be `null` or carry only non-conclusion content such as scope, source, caveat, definition, or action note.
- Follow `configurable-zero-to-one-flow.md` for the implementation order and `configuration-field-reference.md` for field meanings. Do not duplicate the full handoff field list in this routing gate.
- `pageLayoutConfig` must include `layoutSectionMap`, a 12-column `layoutRows` audit, and readable coordinates: every section is an exact `12*K` row group, section row counts sum to page `N`, local section A/B previews are globally disambiguated in final machine rows, every row is exactly 12 cells, no row is longer than 12 cells, total row count `N >= 8`, every block is rectangular, every visible top-level block has `rowSpan >= 3`, every block id maps to one widget and one `blockAreaConfigMap` row, every span records `colStart`, `colSpan`, `rowStart`, and `rowSpan`, every visible block has a unique `blockCoordinate`, and every component slot has a unique `slotCoordinate`.
- Template filter/control design must be explicit before implementation: `filterSurfaceMap`, `toolbarActionMap`, `pillAreaConfig`, and `interactionBehaviorMap` must show visible placement, owner, `filters[].scope`, affected data/API params, affected slot coordinates, refresh scope, and state response for every page filter, business switch, date/period control, block pill, toolbar action, drilldown, jump, modal, drawer, popup, cross-filter, export, refresh, or fullscreen action. For block pills, `pillAreaConfig` must state whether each pill is visual-only or drives `titlePills[].filters`, `titlePills[].params`, `titlePills[].props`, `titlePills[].dataBinding`, or `titlePills[].actions`; metric/period/mode/scenario pills are data-affecting by default.
- `componentExampleConfigMap` must include one row per declared `3 componentArea` slot with `slotCoordinate`, slot pattern code, component slot size, visual-type size compatibility evidence, registered component example ID, standalone Vue file/component id, source/target path, sample/source evidence, props, data, `filterScope`, `dataBinding`, actions, renderer, state, and interaction contract. Use `component-examples/config.ts`, the selected template catalog, and `widget-config-schemas.ts` to select IDs. A slot with only text/prose, placeholder content, a `visualType`, or an inline widget object is unfilled until it references a registered component example ID or a registered custom fallback template and passes size compatibility.
- If the PRD or design-thinking output supplies `displayTheme` and selected pattern cards, use them as scenario evidence for template routing, but do not choose a template from display theme alone. Content volume, navigation depth, interaction density, and display environment still decide.
- If the PRD or design-thinking output supplies a report decision anti-AI result, carry `reportDecisionRisk`, `RPT-*` gaps, metric tree, data story, realistic data needs, linkage/action/trust details, and report-designer requirements into template routing. If it is missing for a report surface, record a routing gap instead of assuming a generic dashboard template is sufficient.
- Bundled templates use Vue 3 + TypeScript + Vite + Element Plus + ECharts + axios as one integrated default stack, with AntV S2 added only for S2-class analytical tables. This is a `MUST/fail` stack contract, not a loose recommendation: do not ship a Vue-only project, a Vue + ECharts project without Element Plus, or a Vue + Element Plus project that hand-draws standard charts instead of using ECharts.
- Template validation must prove the stack contract: `package.json` keeps the required dependencies, `src/main.ts` bootstraps Vue 3 through `createApp`, registers Element Plus globally, imports Element Plus styles, and standard chart visualTypes have an actual ECharts runtime import/wrapper or documented custom-diagram exception.
- Bundled templates must keep `npm run visual:geometry -- --url <url>` available for runtime visual geometry discovery. It uses Playwright to capture screenshots and produce `visual-check/geometry-report.json` with deterministic `VIS-*` findings for clipped content/text, sibling overlap, squeezed repeated rows/cards, list rows where `scrollHeight > clientHeight + 1`, hidden list overflow, axis chart containers below width/height floors or squeeze-risk bands, chart body/plot squeeze, chart/table crowding, and SVG label overlap.
- Bundled template configs must keep list/chart geometry contracts machine-readable. List-like widgets declare `rowHeightPx`, `visibleRowCount`, and `overflowStrategy`; compact/internal action/task lists show at most `2` rows and route overflow to detail/tooltip/drawer/view-all/table fallback. Full line/bar/combo axis widgets estimate post-layout `contentW = contentWidth / 12 * cols - cellPadding * 2` and `contentH = rowHeight * rows - cellPadding * 2`, require `contentW >=300px`, `contentH >=200px`, and `chartBodyH >=180px`; if they cannot meet the floors, they either expand/split the block or explicitly use `visualType: 'compact-sparkline'` / `chartMode: 'compact-sparkline'` with legend, Y-axis unit/name, and permanent labels hidden. Blocks below `400px` wide or `250px` high, or below `500px` with dense/long x-axis labels, must declare `squeezeStrategy`, `axisLabelStrategy`, `dataZoomStrategy`, or an equivalent fallback; `axisLabel.interval: 0` alone is not a valid dense-label remedy. Pie/donut/rose/proportion and sunburst widgets default visible category/hierarchy legends on the right and declare `minAngle` or tiny-slice aggregation/detail fallback when applicable; radar widgets show right-side legends for multi-series comparison and hide single-series legends by default.
- In bundled templates, standard chart widgets must use ECharts through an actual chart component/wrapper and data-driven `option`/`series`. Ordinary Cartesian ECharts charts such as line, bar, scatter, quadrant, and combo default multi-series `legend` to top center when data exists and space permits; single-series or single-metric legends are hidden by default unless an explicit `legendVisible: true` exception is documented. Proportion/pie/donut/rose and sunburst charts default visible category/hierarchy legends on the right; radar charts show right-side legends for multi-series comparison and hide single-series legends by default. Axis charts configure units through `yAxis.name`/axis-unit metadata, keep `yAxis.axisLabel` raw numeric without repeated unit suffixes, compute dynamic Y-axis ranges for NPS/score/rate/current-vs-target charts, use compact `grid.top/right/bottom/left`, place Y-axis titles on left/right sides, place X-axis titles at the bottom, and set target/reference label position to `insideEndTop`. Do not implement standard charts by importing ECharts but drawing the bars, lines, pies, gauges, maps, sunburst arcs, treemap rectangles, Sankey ribbons, tree nodes/connectors, relation graph nodes/edges, axes, or legends with hand-authored SVG/HTML/CSS/canvas. Gauge widgets must use ECharts `series.type: 'gauge'` and carry one bounded metric, min/max, current value, unit, target/threshold/status semantics, overflow behavior, center value, and exact tooltip evidence, not fixed decorative arcs or needles. Path/user/process path widgets must be driven by declared step/link data, not fixed decorative journey lines. Sankey widgets must be driven by node/link data with `source`/`target`/`value`, layer/stage order, metric unit, Top N/`其他`, flow-width mapping, hover path highlight, and exact node/link tooltip evidence, not fixed decorative ribbons. Treemap/rectangular tree map widgets must be driven by hierarchy data, a non-negative additive area metric, Top N/`其他`, label thresholds, color semantics, breadcrumb/drilldown when deep, and exact path/value/share tooltip evidence, not fixed decorative rectangle mosaics. Sunburst widgets must be driven by hierarchy data, a non-negative additive angle metric, visible depth/ring-width budget, Top N/`其他`, sector label thresholds, center content, color semantics, breadcrumb/drilldown when deep, and exact path/value/total-share/parent-share tooltip evidence, not decorative multi-ring pies. Tree/hierarchical tree widgets must be driven by root/parent-child data, visible-depth/expand-collapse state, and Top N/`+N` aggregation, not fixed decorative org-chart boxes.
- If a selected template has `nav[]`, every nav item must map to a substantial business page. Do not ship a navigation template with only the homepage populated.
- Requirement-document shell details must be adapted into the selected template's existing title/logo, `nav`/`page`, `filters`, controls, toolbar, and action slots. Do not add duplicate shell structures that fight the template.
- When a selected template already owns filter invocation, filter design means editing `filters[]`, `filters[].source`, active-state display, and widget/API/resolver bindings. In mock API mode, do not keep static `filters[].options` as fallback. Do not create a standalone filter toolbar, persistent filter bar, or extra filter drawer inside the report development flow.
- Before rendering any refresh, download/export, copy/share, period/date, filter, or toolbar action, declare `controlOwnership` or an equivalent ownership matrix. Default ownership is `template` for page/global filters, refresh, download/export, topbar actions, title/logo/navigation, and route toolbar. Component slots may emit events, but configured behavior should live in `componentSlots[].actions` or block `actions` first; shell defaults handle route, external, drawer/drilldown, modal, popover/popup, cross-filter, fullscreen, export, and refresh. Components may self-develop behavior only when listed in `selfDevelopmentExceptionMap` and when the default action model cannot express it. 组件示例 slot fills are excluded from component-owned filters/controls and may expose their optional removable title plus body, including declared internal auxiliary metrics. Do not move shell-owned controls into components inside report development.
- Template `filters[]` is reserved for horizontal constraints that keep the same component schema. Business domain, report theme, management object, subject area, or first-level analysis perspective must use nav/page/route/tab/segment/perspective state when it changes metric names, component set, table headers, metric口径, or domain vocabulary.
- Template data-source filtering uses configurable `emptyFilterValues`. Empty/no-filter values, aggregate row values, and real detail row ids must be separate semantics; a visible label such as 全部 or Total does not make a value safe to use as both a filter sentinel and a row primary key.
- Dynamic perspective-navigation metrics such as percentages, rankings, status lights, satisfaction scores, completion rates, and risk scores must not be stored in `filterData.meta`; use `businessData`, aggregate datasets, API/provider fields, or custom resolvers with declared lineage.
- Detail Table widgets must declare row grain, primary key/object field, default sort, column metadata, visible column priority, search/sort/pagination/export scope, row detail/action payload, local-filter behavior, and states. Use Element Plus/project table for ordinary row-level details; add AntV S2 only for pivot/cross/wide-matrix/frozen analytical table needs.
- Pivot Table widgets must use visualType `pivot`, declare row dimensions, column dimensions, measures, aggregation formulas/functions, rate numerator/denominator rules, subtotal/grand-total rules, hierarchy depth, sort behavior, density fallback, tooltip/drilldown payload, local metric/display controls, and states. Add AntV S2 only when such S2-class analytical behavior is actually implemented; do not use a hand-rolled merged HTML table for a real pivot.
- Complex/grouped table-header widgets must declare a grouped-column contract when visible columns exceed `8` or fields naturally group. Use Element Plus/project grouped columns for ordinary row tables and S2/project analytical headers for pivot/cross/wide matrices. The widget must preserve `columnTree`, leaf field metadata, units/definitions, computed `colSpan`/`rowSpan`, max depth `<=3` by default, fixed multi-level header, frozen row/primary columns during horizontal scroll, component-local filter vs per-column header-filter separation, tooltip definitions, density fallback, and stable states. Do not fake merged headers with absolute positioned labels or decorative color bands detached from data fields.
- Analysis & Insight widgets may use `visualType: 'text-summary'`, but when they are conclusion cards, insight cards, anomaly/risk explanations, attribution/recommendation cards, definition/data-quality/forecast notes, chart annotations, explanatory empty states, permission/no-result/delay notes, or task notes, they must declare `analysisInsightContract` or equivalent props with subtype, `conclusionRuleId` and generated conclusion template when the conclusion is data-driven, evidence, affected object, action/detail path, confidence/definition/source/freshness, local-filter scope, tooltip payload, and states. Do not ship generic "智能洞察" copy or fixed normal-state conclusions without evidence/action/trust context.
- Composite Panel widgets must be implemented as one business widget with an internal layout contract, not as unrelated top-level `layoutRows` cells. Declare `compositePanelContract`, shared topic, analysis sequence, primary child id, child roles/priorities/min sizes, default `2-3` children and normal max `4`, primary visual weight `50-70%`, `contentH >= CH * 0.60`, panel-level local filters, child-only filter exceptions, shared legend/unit behavior, linked hover/click state, detail-preview limit, responsive fallback, and parent/child loading/empty/error/no-permission states. Do not create nested card shells or equal-weight mini dashboard collages inside a widget.

## Quality Gate

- The selected template matches display scenario, navigation depth, density, and fixed/scroll behavior.
- `frameworkTemplateId` points to a bundled template, and `selfDevelopmentExceptionMap` contains only interaction behavior and registered component example entries.
- PRD-driven or multi-step implementation has a current `templateAssetUnderstandingMap` and Template Build Packet, and implemented source files map back to packet section 11.
- `pageLayoutConfig.layoutSectionMap` exists and splits every page into exact `12*K` parts whose row counts sum to `N`; `pageLayoutConfig.layoutRows` passes the exact 12-column row audit, over-12 rejection, minimum-8-row audit, rectangle audit, block id map, and widget/template map. Any proportional-only or prose-only layout handoff is not ready.
- `layoutCoordinateMap` is present and consistent: every visible block has `blockCoordinate`, every `3 componentArea` slot has `slotCoordinate`, and coordinates match the section preview, `layoutRows`, declared component slot order, metric mounting, conclusion rules, and interactions.
- `filterSurfaceMap`, `toolbarActionMap`, `pillAreaConfig`, and `interactionBehaviorMap` are present when controls or interactions exist; missing visible placement, ownership, filter scope, affected data/API params, active pill runtime path, source slot coordinate, action placement, target type, handler mode, refresh scope, or state response blocks readiness.
- Configured prototype source files pass the file-level ledger rule: every changed `.vue`, `.ts`, `.js`, `.mjs`, `.css`, `.json` config/data-source behavior file, validation script, or action/data registry code file has a sidecar ledger that was read before editing and appended after editing. Generic final summaries, commit messages, or delivery indexes do not replace the sidecar ledger.
- Configured template projects pass the project-level ledger check: `npm run ledger:init` ran before the first source edit and `npm run ledger:check` passes before handoff. This prevents silent projects with no `__change_logs__` baseline.
- `docs/prototype-data-summary.md` exists and is current with actual data files, datasets, fields, metric/conclusion inputs, component data binding, filters, interactions, backend interface method contracts, API/model suggestions, gaps, and verification. Missing, placeholder, stale, or method-incomplete data summaries block backend-facing handoff readiness.
- The absence of a `change_logs` directory is not itself a defect. The required file-level code-ledger location is `__change_logs__/<code-file-name>.changes.md` beside each changed scoped source file.
- The selected template is not justified by display theme alone; it also matches content volume, navigation depth, interaction density, and display environment.
- The selected template is not justified by dashboard appearance alone. It must support the report decision path, including metric tree, data story, detail/action/trust areas, realistic data states, and report-designer behavior when applicable.
- Do not choose a template that forces unresolved `RPT-TEMPLATE-LAYOUT`, `RPT-NO-DATA-STORY`, `RPT-STATIC-FILTERS`, `RPT-NO-ACTION`, or `RPT-DESIGNER-SHELL` findings unless the limitation is recorded and the delivery readiness is `partial` or `blocked`.
- Templates with `nav[]` are used only when multiple substantial nav pages are implemented; if only one page can be populated, do not choose or retain a nav template.
- Requirement-document title/filter/navigation/toolbar/export/permission ideas are mapped into existing template slots instead of implemented as duplicate shell layers.
- Component slots carry only 组件示例 data inside `3 componentArea`: the registered component example ID, mapped standalone Vue file/component id, component slot size, visual-type size compatibility evidence, sample/source evidence, props, data, `filterScope`, `dataBinding`, actions, renderer, state, and interaction contract. If `componentSlots[].slotFills` targets parent `titleArea`, `pillArea`, `summaryArea`, filters, controls, descriptions, or help/explanation copy, move that data to the parent 分块配置 config, shell/page config, or a non-slot widget before readiness. Do not remove component-owned `unit` or `auxMetrics` from registered examples that support them.
- `blockAreaConfigMap` is not ready when any visible block lacks `slotCount`, `componentSlotPattern`, or `slotCoordinateList`, or when a `componentExampleConfigMap` row targets a slot outside the declared list.
- Component slots are not ready when they are text-filled or placeholder-filled instead of bound to registered component examples with standalone Vue files, sample/source evidence, component slot size, and visual-type size compatibility evidence.
- Dynamic component slots are not ready when `dataBinding` is missing, when `filterScope` cannot be traced to `filters[].scope`, or when `actions` lack source coordinate, event name, target type, handler mode, state response, and QA case.
- A block area is not ready when it is represented only by fixed-wrapper naming or `componentRegionPattern`. Create the `createBlockAreaConfig` config and slot list first.
- A visible block is not ready when `blockAreaConfigMap` lacks a PRD section 4A `PATH-*` source and the block is not explicitly support/source/export/permission-only.
- A management-facing visible block is not ready when it implements first-viewport judgment, severity, action, trust/source, or review/export behavior without applicable PRD section 4B gate IDs.
- `1-2 pillArea` and `4 summaryArea` may be omitted only after the need check is recorded as not needed, except the built-in conclusion-card summary hiding rule. `1-1 titleArea` and `3 componentArea` require explicit configs for every accepted block. Units and auxiliary information are validated through component example props/config, not through parent block areas.
- A fallback chart/table/list implementation is not allowed to bypass the 组件示例 layer. Custom ECharts work must be packaged as a standalone component example and registered/copied before the slot is considered filled.
- If template filters are present or needed, they are configured through the template's native `filters[]` and existing invocation surface; standalone filter toolbars/bars are rejected.
- Duplicate visible refresh, download/export, copy/share, date/period, global-filter, toolbar, or shell action controls fail as `VIS-DUPLICATE-CONTROL` or `RPT-SHELL-DUPLICATE`. Component interaction controls are allowed only when listed in `selfDevelopmentExceptionMap`, scoped to the component, and not duplicating shell controls. Component-local controls are not allowed inside 组件示例 slot fills; if used in a non-slot business widget, they must prove local scope and must not silently change template `filters[]`, page query, permission, pagination, export, or other widgets.
- Template `filters[]` contains horizontal constraints only. Schema-changing domain/theme/management-object/perspective controls are rejected as ordinary filters unless `componentSchemaImpact: row-scope-only` is explicitly proven.
- Empty-filter behavior is configured through `emptyFilterValues` on data/API sources or a declared equivalent adapter. A hardcoded assumption that `all` always means no filter fails template readiness.
- Aggregate/subtotal/total rows use distinct aggregate row semantics such as `rowRole`, `aggregateValue`, or an adapter-normalized synthetic key. Using `all` as a business primary key for those rows fails unless an upstream legacy contract is isolated and documented.
- `filterData.meta` contains only static dimensional attributes. Dynamic navigation KPIs in `filterData.meta` are rejected unless explicitly marked as static display copy and excluded from KPI/consistency checks.
- Template project structure remains complete after materialization and configuration.
- Template stack structure remains complete after materialization: Vue 3 + TypeScript + Vite + Element Plus + ECharts + axios stay installed and wired, `src/main.ts` keeps Element Plus registration/style imports, and chart visualTypes prove real ECharts runtime ownership.
- Native shell navigation, filters, toolbar, theme, and logo slots are preserved unless explicitly redesigned.
- Parent-block title/body chrome is widget/composite-owned for ordinary non-slot widgets. It must declare `blockChromePattern` or an inherited default and preserve non-slot component-owned title/control areas, chart/table/list floors, and runtime geometry proof. Shell-level title bands, raw copied HTML wrappers, and uninspectable decorative CSS fail template readiness.
- Non-slot component-owned title/control areas preserve readable title and compact local controls. Single component-local filter with `2-4` short values and fit proof uses capsule; single local filter with `>4` values, long labels, or failed width fit uses dropdown; multiple local filter groups use panel trigger; detail actions use lightweight links. The template shell must not render these block-internal titles or controls, and 组件示例 slot fills must not render them either.
- Non-slot `localFilters[]` affect only the current component's already loaded data. They must not replace template `filters[]`, page/global query scope, permission scope, backend aggregation, pagination, export scope, or other widgets, and must not appear inside 组件示例 slot fills.
- Analysis & Insight widgets pass contract validation and implementation proof: subtype/family, conclusion-before-evidence, evidence or insufficiency state, affected object when relevant, action/trust/source/freshness when relevant, local-filter scope, tooltip/detail path, loading/empty/error/no-permission/data-delay states, matching DOM/classes/attributes, CSS behavior, and browser crop/geometry proof when code or URL exists. Generic explanation copy without this contract fails; config-only contract pass is `LAZY-CONTRACT-THEATER`.
- Composite Panel widgets pass contract validation and implementation proof: `compositePanelContract` fields must map to actual child DOM, CSS layout, renderer options, shared legend/unit/filter behavior, linked hover/click state, responsive fallback, and parent/child state handling. Passing only a regex/config validator without runtime or DOM evidence is not template readiness.
- Shared template layout tokens from `template-layout-design-system.md` are reused instead of rediscovered from source code or changed ad hoc.
- Template shell hover/focus feedback for cards, toolbar buttons, nav items, and filter chips preserves geometry. Prefer in-bounds border glow or inset glow; do not use hover translate/scale that can clip borders/shadows in fixed grid or overflow-hidden containers.
- Data completeness is checked before filter binding; missing option rows, missing business rows, missing fields, single-snapshot mock data, or missing resolver/API branches are data gaps and cannot be hidden with `ignoredFilters`.
- Composite block no-data masks pass hierarchy checks: parent mask only when all child sub-blocks are empty; otherwise child masks cover affected child title/control plus component body.
- Validation scripts and start helpers remain available.
- `npm run validate:dashboard` stack-contract failures block `ready`; do not bypass them by running only Vite build, screenshot review, or manual browser checks.
- Runtime visual geometry failures from `npm run visual:geometry -- --url <url>` block `ready` at `major` or above. List-row clipping (`scrollHeight > clientHeight + 1`), hard-hidden list overflow, axis chart container width below `300px`, axis chart container height below `200px`, squeeze-risk containers without strategy metadata, axis chart body below `180px`, plot-height squeeze, and chart/table crowding are direct failures. If Playwright/Chromium is missing, record the exact blocker and keep runtime readiness `partial` rather than reverting to manual screenshot review.
- Template asset paths in workflow or layout skills point to this skill, not to layout skill assets.
- Primary/global filters that should change business data are not placed in `ignoredFilters`; mock/API/resolver data proves at least one affected widget value changes for non-default filter states.
- Standard chart widgets are not accepted when they only import ECharts but draw chart marks with hand-authored SVG/HTML/CSS/canvas. ECharts SVG renderer is allowed only when generated by ECharts options; ordinary bar/line/scatter/combo axis charts require visible top-centered default legend when data exists and space permits, Y-axis unit configuration, raw numeric Y-axis tick labels, compact grid sides, side/bottom axis-title placement, dynamic Y-axis range for bounded NPS/score/rate/current-vs-target charts, target/reference `insideEndTop` labels, and exact value + unit tooltip evidence. Proportion/pie/donut/rose, radar, and sunburst charts require visible right-side default legends unless an explicit compact/small-container fallback or `legendPosition: 'hidden'` reason is recorded. Combo charts require a shared x-axis, ECharts `bar` plus `line`/`markLine` series, paired business relationship, bar/line metric units, left/right axis mapping, visible series limits, category-density fallback, and exact tooltip evidence; Gauge charts require ECharts `series.type: 'gauge'`, one bounded metric, min/max, current value, unit, target/threshold/status semantics, center value, overflow behavior, and exact tooltip evidence; parallel-coordinate charts require `parallelAxis`, `series.type: 'parallel'`, object/sample data, ordered dimensions, per-axis unit/range/direction, line density, highlight/brush, and exact object tooltip evidence; funnel charts require ordered stage data, shared population/cohort logic, value/unit, entry share, stage conversion, drop value/rate, total conversion, tooltip evidence, and either ECharts `series.type: 'funnel'` or a data-driven ECharts horizontal `bar` funnel; path/user/process path charts require declared step/link data and transition evidence, Sankey diagrams require ECharts `series.type: 'sankey'` or a declared data-driven custom exception plus `source`/`target`/`value` links, layer/stage order, Top N/`其他`, flow-width mapping, hover path highlight, and exact node/link tooltip evidence, treemap/rectangular tree maps require hierarchy data, non-negative additive area metric, Top N/`其他`, label thresholds, color semantics, and exact path/value/share tooltip evidence, sunburst charts require hierarchy data, non-negative additive angle metric, visible depth/ring-width budget, Top N/`其他`, sector label thresholds, center content, breadcrumb/drilldown, and exact path/value/total-share/parent-share tooltip evidence, tree/hierarchical tree charts require declared root/parent-child data, expand-collapse, and node evidence, and relation/network graphs use ECharts `graph` unless an explicit custom-diagram exception is recorded.
- Table widgets are not accepted when `visualType: 'table'` is only a generic list of fields. Detail Tables require row grain, primary key, prioritized visible columns, type-specific width/alignment, default sort, table-body row budget, fixed header/frozen-column behavior when needed, provider/API/resolver scope for global search/sort/pagination/export, local-filter limits, row detail/action payload, and stable empty/loading/error/no-permission states.
- Pivot widgets are not accepted when `visualType: 'pivot'` is only a decorative matrix or raw rows reshaped as columns. They require row dimensions, column dimensions, measures, aggregation formulas/functions, subtotal/grand-total rules, rate numerator/denominator recomputation, S2/project analytical renderer, frozen headers, row/column density fallback, tooltip/drilldown payload, and stable empty/loading/error/no-permission states.
