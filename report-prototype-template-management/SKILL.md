# Report Prototype Template Management

## Stage Scope

Classification: 原型阶段.

Use this copy only inside any compatible prototype agent bundle. Treat technical solution, backend, frontend delivery, and testing work as downstream handoff artifacts or blockers, not as implementation steps to execute from this workflow.

Agent-neutral rule: this workflow is a capability contract, not a platform-specific instruction file. Capability IDs such as `report-visual-layout-design`, `report-design-system-governance`, and `code-change-ledger-management` are stable routing names; Claude, Hermes, Codex, or another agent must map them to equivalent local prompts, tools, workflow nodes, or documentation before execution.

## Positioning

Use this capability when the task needs a runnable report prototype template or must modify a bundled template. It owns template asset selection, configuration strategy, extension points, validation scripts, local startup helpers, and template-specific implementation boundaries.

It does not own report-path business logic, component visual rules, API documentation, or production frontend integration. Report-path and component-mapping decisions belong in `report-prd-document-generation` through `references/report-type-implementation-patterns.md` and `references/component-mapping-prd-contract.md`; component visual, delivery, and runtime concerns route to `report-component-style-design`, `report-delivery-pipeline-governance`, and `frontend-runtime-qa-validation` as appropriate.

Default routing: use the latest configurable report project flow for runnable report prototypes. Select one bundled framework template, configure `pageLayoutConfig`, create `blockAreaConfigMap`, declare component slots, bind registered component examples, and validate release readiness. In the report development flow, custom/free shell, exact shell restoration, existing shell preservation, custom page layout, custom block layout, duplicate filter/nav/toolbar surfaces, static shell paths, and blank new Vue projects are blockers or out-of-scope exceptions, not implementation routes. The only self-developed surfaces are interaction behavior and registered component examples. PRD wording or source artifacts that mention HTML are evidence only and must not switch the template route.

Default development stack contract: bundled report prototype projects use `Vue 3 + TypeScript + Vite + Element Plus + ECharts + axios` as one integrated stack. This stack contract describes what the selected template preserves; it does not mean starting from an empty Vue3 engineering project. Do not downgrade this to "Vue 3 only", "Vue 3 + ECharts only", or a hand-built UI shell. Element Plus owns base controls, messages, tables/forms/dialog-ready styling, locale, and CSS variables; ECharts owns standard chart rendering through actual option/series/runtime instances. Add AntV S2 only for pivot/cross/wide analytical table needs.

Latest configurable-flow rule: every new report project must start from `references/configurable-zero-to-one-flow.md`. The active chain is `frameworkTemplateId -> pageLayoutConfig -> filterSurfaceMap -> toolbarActionMap -> interactionBehaviorMap -> blockAreaConfigMap -> componentSlotConfigMap -> componentExampleConfigMap -> customEChartExampleMap when needed -> releaseValidation`. Block areas are created with `createBlockAreaConfig` and exposed through `blockAreaConfigMap`; the runtime block implementation is `BaseLayoutSpan.vue`. Component slots are filled only by registered `componentExampleId` values from `src/widgets/templates/component-examples/config.ts`. Existing component examples bind to filters through `componentSlots[].data`, `componentSlots[].filterScope`, and `componentSlots[].dataBinding`; page interactions bind through `componentSlots[].actions` or block `actions`, with shell defaults for route, external, drawer/drilldown, modal, popover/popup, cross-filter, fullscreen, export, and refresh. When no existing example fits, self-development means registering a new component example or using `component-example-catalog:custom-echart-component-template`.

Zero-size-design responsibility rule: project configuration expresses business semantics, not pixel fitting. The model configures which registered component example fills each slot, what data/API/filter/interaction it consumes, and what business note appears in the block summary area. It must not hand-tune slot height, title height, summary height, font ratios, row ratios, padding, or overflow to make a page fit. Template runtime, registered component examples, `validate-dashboard-contract`, and `visual-geometry-audit` own automatic fitting. If a registered example cannot fit a declared slot, the model must choose another registered example or create/fix a registered component example with an auto-fit contract; it must not patch per-page size values as the solution.

Built-in prototype API rule: bundled templates include `npm run mock:api` and `npm run dev:mock`. Use this npm mock API path when the user needs backend-shaped data without production backend implementation. Put option rows and mock business rows in `src/data/dashboard.dataset.json`, expose them through `/api/filter-options/*` and `/api/report/*`, configure `filters[].source` plus `componentSlots[].data` / owning widget `data` with `id: 'apiData'` or `id: 'httpData'`, and document the response shape in `docs/mock-api-contract.md` and `docs/prototype-data-summary.md`. The data summary is a replacement-first and metric-mapping-first backend handoff: it must include a Mock API To HTTP API Replacement Matrix, Metric To Interface And Source Mapping, concrete service methods, HTTP paths, request DTOs, response DTOs, frontend consumers, data binding/adapter mapping, source table/field/formula hints, permission/cache/error notes, and mock-to-real replacement guidance. Do not create component-local data loaders or TypeScript fixture modules for rows.

Mock API runtime stability rule: when a template uses `apiData` / `httpData`, start and verify with `npm run dev:mock` or `npm run mock:api` plus Vite proxy, not plain `npm run dev`. Before declaring runtime ready, smoke `/api/health`, one `/api/filter-options/*` endpoint, and one `/api/component-props/:componentDataKey` endpoint. The shared request client must not spam repeated generic `Request failed` messages; it should rate-limit identical user-facing messages, distinguish mock/API unavailable, 404, 429, and business-code failures, and keep URL/status details in `console.warn` for diagnosis.

Bundled-template handoff hygiene rule: template assets must feel like a new project. Before shipping or handing bundled templates to users, keep `DELIVERY_INDEX.md` and `__change_logs__/*.changes.md` in initialized baseline form, remove historical patch/snapshot evidence files from `__change_logs__/patches` and `__change_logs__/.snapshots`, and keep only user-facing docs such as `README.md`, `docs/mock-api-contract.md`, and `docs/prototype-data-summary.md` as substantive handoff documents. Detailed iteration histories belong in copied/configured projects after user-specific edits, not in the reusable template baseline.

## Anti-Laziness Gate

For non-trivial work, apply `quality-gate-validation` `references/anti-laziness-execution-gate.md` before final output, handoff, or readiness. Do not mark the result ready while `LAZY-*` findings remain open, when available local evidence was not inspected, when owning capabilities were skipped, or when proof is limited to generic statements such as "checked", "optimized", "looks good", or "implemented".

## Workflow

0. Enforce the configurable zero-to-one report flow. Framework template, page layout config, block area configs, block-level title/pill/component/summary areas, navigation, filters, toolbar, export, permission, and shell state use the bundled template contracts. Record self-developed items only when they are registered component examples or interaction behaviors.
0A. Enforce the runnable stack contract. Preserve the selected template's Vue 3 + TypeScript + Vite + Element Plus + ECharts + axios stack. A PRD, requirement document, or HTML source sample is requirement evidence, not output-format authority.
0B. Enforce configuration-first implementation. Select the template asset root through `frameworkTemplateId`, then configure layout, block areas, slots, component examples, data, interactions, and validation rows before source edits. Do not use an independent project bootstrap or static shell inside this configurable report flow.
0C. For PRD-driven or multi-step template implementation, read `references/template-asset-construction-contract.md`, create `templateAssetUnderstandingMap`, then create or validate a Template Build Packet from `references/template-build-packet-contract.md` before source edits. Treat it as the single construction plan: every page, block, slot, filter, interaction, data/API row, conclusion rule, exception, target file, and validation command must be represented there before implementation consumes it.
1. Select exactly one 框架模板 using display theme, selected pattern cards, content volume, navigation depth, interaction density, and display environment.
2. Design 页面布局配置 through `layoutSectionMap`, `layoutRows`, stable block ids, readable `layoutCoordinateMap`, PRD section 4A `PATH-*` source per visible block, PRD section 4B gate ID traces when applicable, and page/nav widget wiring; for templates with `nav[]`, define substantial nav-page content before copying or editing. Split each page into readable `12*K` parts before raw rows, such as section 1 `12*3`, section 2 `12*3`, section 3 `12*3`; each section records business purpose, local row preview, global row range, and block coordinates. `layoutRows` must be `12 * N` with exactly 12 cells per row, no row longer than 12 cells, and `N >= 8`; every visible top-level block uses row span `N >= 3`. Local section A/B previews may repeat for readability, but final machine rows must disambiguate blocks. Use `blockCoordinate` as `R-B` and `slotCoordinate` as `R-B-S`; for example, when section 2 has two `6*3` blocks, the second block is `2-2`, and its first component slot is `2-2-1`.
3. Based on 页面布局配置, create one block area config for every page block with `createBlockAreaConfig` and expose page groups through `blockAreaConfigMap`. Each block row must carry `blockCoordinate`, the source `RTP-*` / `PATH-*` step, applicable `ESG-*` / `SEV-*` / `ACT-*` / `TRUST-*` / `MEET-*` IDs unless it is support/source/export/permission-only, plus `slotCount`, `componentSlotPattern`, and `slotCoordinateList`. Slot patterns such as `A`, `AB`, `ABC`, or `AAB|CCD` describe slots inside `3 componentArea`, not page `layoutRows`. Page `layoutRows` decides block size, and `BaseLayoutSpan.vue` is the only runtime block area implementation. Every block uses only the standard block-level areas: `1-1 titleArea` 标题区, `1-2 pillArea` 胶囊按钮区, `3 componentArea` 组件区, and `4 summaryArea` 说明区. Do not confuse these internal area codes with page coordinates; standard area targets should use `blockCoordinate + areaName`, such as `2-2:pillArea`.
   - Before selecting SingleSlot or MultiSlot, analyze the user's question, conclusion count, evidence relationship, comparison needs, and density. Choose SingleSlot when one conclusion card/component should dominate the block; choose MultiSlot only when parallel evidence, conclusion-card-plus-driver, comparison, or tightly related components are genuinely needed.
   - In MultiSlot templates, put the conclusion card or primary conclusion component in the first component position whenever the requirement uses componentized conclusions. Use later slots for evidence, drivers, details, actions, or trust/source support. Text-only or narrative conclusions belong to `4 summaryArea` only when the same block has no conclusion card/component. When a block contains `ConclusionExampleCard`, hide the outer block `4 summaryArea` with `showSummary:false` and `summaryHiddenReason`; keep the conclusion card's internal supplemental/evidence/action sections intact. Every conclusion target must bind `conclusionRuleId` from PRD `conclusionRuleMap`.
   - `1-1 titleArea` and `3 componentArea` are always required. `4 summaryArea` is displayed by default and should use the block `note` when no separate `bodySummary` is provided. The built-in exception is a block that contains `ConclusionExampleCard`: that block hides the outer summary area because the card owns the conclusion, evidence, action, and supplemental note surface. Other hidden summary areas require explicit user intent plus `showSummary: false` and `summaryHiddenReason`. `1-2 pillArea` is optional and should be omitted or recorded as `null` when the requirement does not need it. Units and auxiliary information belong only inside registered component examples that declare those props; they are not block-level areas.
4. Configure the 分块配置 `1-1 titleArea`: implement the title and title style.
5. Decide whether `1-2 pillArea` is needed. Configure pill buttons when needed; otherwise record `pillAreaConfig: null` with `notNeededReason`. If the PRD includes metric switch, period/month-year switch, local mode switch, scenario switch, or block-level compact control, it must be visible in `1-2 pillArea` unless it belongs to the template-native filter surface or toolbar. Data-affecting pills must configure `titlePills[].filters`, `titlePills[].params`, `titlePills[].props`, `titlePills[].dataBinding`, or `titlePills[].actions`; the runtime exposes them through `$context.activeTitlePillId`, `$context.activeTitlePillLabel`, and `$context.activeTitlePill`.
6. Based on the declared component slots, choose registered component examples from `src/widgets/templates/component-examples/config.ts` for `3 componentArea` / `componentSlots`. Every slot must carry `slotCoordinate`, slot pattern code, component slot size, `componentExampleId`, Vue component/export/registry evidence, visual type, props/config/data/state contract, `filterScope`, `dataBinding`, and `actions` when it participates in configured interactions. If no suitable example exists, use `component-example-catalog:custom-echart-component-template` or self-develop a new Vue component under `src/widgets/templates/component-examples/`, export it from `component-examples/index.ts`, add its schema in `component-examples/config.ts`, and register it in `widgets/registry.ts` before filling the slot. Do not attach block-level title/pills, filters, controls, description, explanation, or summary copy to the component slot. Component props may include component-owned fields such as `unit` and `auxMetrics` when the registered example defines and renders its own internal unit area or auxiliary metric strip. Chart-like component examples that expose component-owned `auxMetrics` must separate two configuration meanings: `config.layout.orientation` controls how the auxiliary strip and chart body sit together, while `config.aux.orientation` controls whether the metric items inside that strip are `horizontal`, `vertical`, or `auto`. Explicit `config.aux.orientation` wins; only `auto` may adapt to available space. By default, the auxiliary information area sits between the component title strip and the chart/table/list body; horizontal component layout uses a horizontal strip with key/value stacked vertically, and vertical component layout uses a vertical strip with key/value on one line. Component example title visibility is automatic and not a manual design choice: single-slot blocks must hide the component short title and use the parent block title; multi-slot blocks must show the component short title for every slot, with no multi-slot hidden-title exception. Registered component examples must implement an auto-fit contract for their internal title, unit, auxiliary metrics, value, chart, table, list, accessory, and overflow behavior. Data item display-name fields must be explicit: `label` is preferred for generic list/ranking/action items, but registered examples should normalize common backend dimension names such as `name`, `regionName`, `region`, `areaName`, or `dimension` when those are natural API fields. Do not accept a list/ranking widget whose values render but names are blank. For KPI, target-progress, scorecard, conclusion-card, and other core-value component examples, include component-internal fit proof before accepting the slot: the main value/text, unit/suffix, accessory row, spark/reflection layer, and optional internal title strip must fit inside the slot at the target viewport. `ConclusionExampleCard` core conclusion content must be vertically centered inside the core area while its title/eyebrow and supplemental/evidence/action rows keep their own alignment. If an internal title is hidden in a single-slot block, the component must release the title-row height: the hidden header must not leave a reserved grid/flex row, and the body/content must explicitly occupy the full available component area instead of being auto-placed into the hidden title row. If the proof is not clean, choose another registered example or fix the component example auto-fit behavior.
7. Configure `4 summaryArea`: it is visible by default and should carry the block `note` or a separate `bodySummary`. When no conclusion card/component exists, it may carry a data-driven text-only/narrative conclusion through `summaryAreaConfig.conclusionRuleId`, or a static note/caveat/explanation that is not a business conclusion. When the block contains `ConclusionExampleCard`, hide the outer block summary area and keep the card's internal supplemental/evidence/action surface. Other summary hiding requires `showSummary: false` plus `summaryHiddenReason`.
10. Carry the layout-provided `blockChromePattern` matrix or explicitly inherit `template-default`/`plain-enterprise` for every styled parent block before widget body content is configured. Block chrome is widget/composite-owned and must not create a duplicate shell title band.
11. Verify report-decision compatibility: metric tree/data story, detail/action/trust capacity, realistic data states, and report-designer needs when applicable.
12. Materialize or update the selected template project target without changing its framework shell or stack contract.
12V. Before editing an existing configured project, use `delivery-version-management` to read or initialize the root `DELIVERY_INDEX.md`; consume its latest change history, stale rows, gaps, and next-change notes before planning source or config edits.
12A. In the configured project, run `npm run ledger:init` before source edits when no baseline exists, creating `__change_logs__` sidecars for scoped source files.
13. Keep shell-owned behavior in template config/data/action registries and widget registries.
14. Preserve native template filter surfaces; configure `filters[]`, `filters[].scope`, data sources, empty-filter values, resolvers, and widget/component-slot bindings instead of adding a duplicate filter bar. Produce a `filterSurfaceMap` that shows every visible filter/control, option source, default, scope, affected blocks/components/slot coordinates, data/API params, and refresh response. For existing component examples, bind filters through `componentSlots[].data.requiredFilters`, API/query params, `filterScope` matching `filters[].scope`, and `dataBinding`; do not create component-example-specific filter panels.
14A. When lightweight backend-shaped data is needed in the prototype, prefer the template npm API path: `npm run dev:mock` starts the mock API plus Vite proxy, `scripts/mock-api-server.mjs` reads `dashboard.dataset.json`, and standard configs use `api.responsePath` (`data.items` for filter options, `data.rows` for component rows). In mock API mode, do not keep static `filters[].options`, static widget rows, chart series, KPI values, or component demo props as runtime fallback data. Component-example slots should use `/api/component-props/:componentKey` by default and map `rows[0].props` through `dataBinding.propsObjectField: 'props'`. Runtime QA must confirm the API is available; repeated generic request-failure toasts are a blocker because they hide whether the issue is missing mock API startup, endpoint mismatch, throttling, or data contract failure.
15. Declare control ownership before adding UI controls. Default: template owns page/global filters, refresh, download/export, topbar actions, logo/title/navigation, and route-level toolbar. Page interactions must be configured with `actions` on the block widget or component slot first; shell defaults can execute route, external, drawer/drilldown, modal, popover/popup, cross-filter, fullscreen, export, and refresh. Use `src/actions/registry.ts` only for custom permission, API orchestration, analytics, host integration, or business-specific overlay behavior. Do not move shell-owned controls into components inside the report development flow. Interaction configuration is not ready until a runtime click/key activation has been verified to produce the expected route, drawer, modal, popover, cross-filter, refresh, fullscreen, export, or external-open state.
16. Preserve the stack contract before source edits: keep `vue`, `@vitejs/plugin-vue`, `vite`, `typescript`, `vue-tsc`, `element-plus`, `echarts`, and `axios` in package dependencies; keep `src/main.ts` bootstrapping Vue 3 and registering Element Plus; use ECharts for standard chart widgets instead of manual SVG/HTML/CSS/canvas marks.
17. Before editing configured template source, read/create the sidecar code ledger through `code-change-ledger-management`; append version entries after edits.
18. Validate chart/table/component fidelity through owning component references when widgets are added or changed.
18A. After data, filters, widgets, generated conclusion rules, and interactions are configured, generate or update `docs/prototype-data-summary.md` in the copied prototype project using `references/prototype-data-summary-contract.md`. The document must summarize actual data files, data modes, datasets, fields, metric/conclusion inputs, Metric To Interface And Source Mapping, component binding, filter semantics, interaction payloads, a Mock API To HTTP API Replacement Matrix, backend interface method contract, backend API/model suggestions, gaps, and verification. Treat this as the final backend handoff artifact: every displayed metric, KPI, chart/table measure, ranking value, generated conclusion variable, target/reference value, and derived status must map to metric name/id, interface/service, response field, source table/view/model, source columns, formula, grain, dimensions, filters, and related `GAP-*`; every mock-backed filter, summary/conclusion variable, chart, table, KPI, list, interaction, and export surface must state the current mock endpoint, mock request/response shape, component-required shape, target HTTP API, request DTO, response DTO, direct-vs-adapter replacement decision, config/code change, permission/cache/error notes, and related `GAP-*`; every backend method row must include suggested service method name, HTTP path, request DTO, response DTO, frontend consumers, binding/adapter mapping, permission/cache/error notes, and mock-to-real replacement guidance.
19. Run `npm run ledger:check`, template validation, build, and dev/preview startup when a local URL is required.
19V. After edits and validation evidence are known in a copied/configured user project, use `delivery-version-management` to append the current change to `DELIVERY_INDEX.md`, including changed files, impacted pages/blocks/slots/components, data/API/filter/conclusion impact, validation, `docs/prototype-data-summary.md` status, code-ledger status, and next-change notes. For bundled reusable template assets, reset `DELIVERY_INDEX.md` and file ledgers to initialized baseline form before handoff so users start from a clean project history.
20. When widgets are configured, `npm run validate:dashboard` must check list and chart geometry contracts: list-like widgets declare `rowHeightPx`, `visibleRowCount`, and `overflowStrategy`; compact/internal action lists show at most `2` rows and use detail/tooltip/drawer/view-all/table fallback. Full line/bar/combo axis charts first estimate post-layout `contentW/contentH` from grid span, `contentWidth`, `rowHeight`, and `cellPadding`, then require `contentW >=300px`, `contentH >=200px`, and `chartBodyH >=180px` unless they explicitly use `compact-sparkline` with legend, Y-axis unit/name, and permanent labels hidden. Blocks under `400px` wide or `250px` high, and blocks under `500px` with dense/long x-axis labels, must declare squeeze/dataZoom/axis-label/fallback strategy. ECharts Cartesian widgets keep compact `grid.top/right/bottom/left`, side-placed Y-axis titles, bottom X-axis titles, `insideEndTop` target labels, dynamic NPS/score/rate Y-axis ranges, and top-centered legends for multi-series data when space permits; single-series or single-metric legends are hidden by default unless an explicit `legendVisible: true` exception is documented. Pie/donut/rose/proportion and sunburst widgets default visible category/hierarchy legends on the right and declare `minAngle` or tiny-slice aggregation/fallback where applicable; radar widgets show right-side legends for multi-series comparison and hide single-series legends by default.
21. When a runnable URL exists, run `npm run visual:geometry -- --url <url>` and include `visual-check/geometry-report.json` so clipped content, text overflow, sibling overlap, squeezed rows/cards, list rows where `scrollHeight > clientHeight + 1`, hidden list overflow, narrow/short chart containers, chart body/plot squeeze, chart/table crowding, SVG label overlap, missing multi-slot titles, and summary-area overflow are discovered automatically before readiness. The audit must inspect real chart panes/ECharts instances, not broad selectors such as `[class*="chart"]`; axis-chart checks apply only to token-matched Cartesian chart types such as line/bar/combo/column and must explicitly exempt non-axis charts such as proportion, radar, funnel, sunburst, and heatmap. Also take or inspect a local screenshot crop and DOM geometry for every KPI/core-value block. Any `VIS-MULTI-SLOT-TITLE-MISSING`, `VIS-TEXT-CLIPPED`, `VIS-CONTENT-OVERFLOW`, or visible clipping inside `.layout-slot-content-widget`, `.placeholder-cell-body-section-3`, `.kpi-example-card`, `.target-progress-example-card`, `.metric`, `.score`, `.value`, or `.card-value` is a readiness blocker even when the generic visual script reports it as `minor`.

## Quality Gate

- Do not choose custom shell/page/block/supporting-area development inside the report development flow. Framework shell, page layout, block area runtime, block-level title/pill/component/summary areas, navigation, filters, toolbar, export, permission, and shell state must use templates; only interaction behavior and registered component examples may be self-developed. Component-level title/unit/auxiliary information/chart/table/list areas must come from registered component examples, not from block-level custom areas.
- Do not choose static HTML output or a hand-built HTML shell because the PRD, requirement document, attachment, or source sample mentions HTML. Runnable report prototypes use the selected configurable template project and preserve Vue 3 + TypeScript + Vite + Element Plus + ECharts + axios.
- Do not create a new Vue3/Vite project inside this flow. Runnable report prototypes use the selected configurable template project; self-development is limited to registered component examples and interaction behaviors.
- Do not mark template configuration ready when a visible block lacks a PRD section 4A `PATH-*` source, unless it is explicitly support/source/export/permission-only.
- Do not mark template configuration ready for management-facing reports when visible first-viewport, severity, action, trust/source, or review/export blocks lack required PRD section 4B gate IDs.
- Do not mark the template implementation ready until it satisfies the ready gates in `references/configurable-zero-to-one-flow.md`; do not duplicate that step chain in this entrypoint.
- Do not mark PRD-driven or multi-step template implementation ready when the Template Build Packet is missing, blocked, stale, or not consumed by source edits and validation evidence.
- Do not mark PRD-driven or multi-step template implementation ready when `templateAssetUnderstandingMap` is missing, stale, or lacks actual asset evidence from the selected template's block area runtime, component example catalog, widget registry, and validator.
- Do not accept any non-configurable block implementation or a `componentRegionPattern` alone as the report block implementation. The selected block must be represented by `createBlockAreaConfig` config and rendered by `BaseLayoutSpan.vue`.
- Do not choose a nav template unless multiple substantial nav pages are implemented.
- Do not mark ready when a `4 summaryArea` conclusion, conclusion card, or `analysisInsightContract` renders fixed normal-state business copy instead of consuming a PRD `RULE-*` row through `conclusionRuleMap`. Static copy is allowed only for source/scope/caveat/definition/action notes or declared empty/permission/insufficient-data fallbacks.
- Do not accept a generated/configured project that keeps Vue 3 but drops ECharts or Element Plus. Missing `echarts`, `element-plus`, Vue 3 bootstrap, Element Plus global registration/style import, or actual ECharts runtime ownership for chart widgets is a template readiness failure.
- Do not replace Element Plus controls with ad hoc HTML controls for ordinary buttons, selects, popovers, drawers, dialogs, tables, messages, or form-like controls unless the user names an existing project design system and the control is inside an allowed interaction or registered component example exception.
- Do not add duplicate shell, filter bar, toolbar, or navigation layers over existing template slots. Redesign requests for those surfaces are blockers or out-of-scope exceptions for report development.
- Do not fill component slots with block-level title pills, filters, controls, description/help text, or summary/explanation copy. Component slots carry only the selected registered component example and its props/config. Component-owned internal areas such as title, unit, `auxMetrics`, chart/table/list body, and compact overflow behavior are part of the registered component example contract.
- Do not accept text, prose, placeholder copy, `visualType` alone, or an inline widget object as a filled component slot. A slot is filled only when it references a registered `componentExampleId` whose Vue component is exported and registered, or a custom ECharts component example that has already been registered.
- Do not accept `pageLayoutConfig` unless every `layoutRows` row is exactly 12 cells, no row exceeds 12 cells, total rows are at least 8, every block is rectangular, and every block id maps to one block area config plus one component slot config set.
- Do not accept `pageLayoutConfig` unless `layoutSectionMap` exists, every section is exact `12*K`, section row counts sum to page `N`, and local section preview letters are disambiguated in final machine rows.
- Do not accept `pageLayoutConfig`, `blockAreaConfigMap`, or `componentExampleConfigMap` when any visible block lacks `blockCoordinate`, any component slot lacks `slotCoordinate`, or coordinates are duplicated/inconsistent with the page preview, `layoutRows`, block area slot order, metric mounting, conclusion rules, or interaction sources.
- Do not accept `blockAreaConfigMap` or `componentExampleConfigMap` when any block lacks `slotCount`, `componentSlotPattern`, or `slotCoordinateList`, or when any component example row maps to an undeclared slot.
- Do not accept template implementation when `filterSurfaceMap`, `toolbarActionMap`, `pillAreaConfig`, or `interactionBehaviorMap` is missing for a control that changes data, navigation, export, drilldown, modal/drawer/popup state, metric set, date/period, or business-line scope. If a block has no pills, `pillAreaConfig` must include `notNeededReason`.
- Do not accept a mock-API-backed template run from plain Vite dev when component/filter data depends on `/api/*`. Missing `npm run dev:mock`, failed `/api/health`, or repeated generic request-failure toasts means runtime readiness is `partial` or `blocked`.
- Do not accept configuration that solves fitting by hand-tuning per-page slot height, title height, summary height, font ratios, row ratios, padding, or overflow. Fitting belongs to template runtime and registered component example auto-fit behavior; configuration chooses semantic component/data/filter/interaction bindings.
- Do not accept a visible block with an omitted `4 summaryArea` unless the block contains `ConclusionExampleCard`, or the user explicitly requested no notes/explanations and the config records `showSummary: false` plus `summaryHiddenReason`. A conclusion-card block must keep the card's internal supplemental/evidence/action sections; only the parent block summary area is hidden.
- Do not accept multi-slot blocks that hide component example short titles. Single-slot blocks hide component short titles and rely on the parent block title; multi-slot blocks must show the short title for every slot so parallel evidence remains distinguishable.
- Do not accept component examples that introduce duplicate shell chrome, block titles, global filters, or toolbar controls. A component example may have only component-local body UI and optional compact internal title when needed for multi-slot disambiguation.
- Do not accept KPI/core-value component examples whose visible value, suffix/unit, accessory row, sparkline, or reflection layer is clipped, vertically offset, or squeezed by a hidden-but-still-reserved internal title row. A parent block and its slots can be perfectly aligned while the component content is still failing; inspect component-internal rectangles and screenshot crops before marking ready.
- Do not accept any component example where a hidden single-slot internal title leaves content in a collapsed `20px` title row or another hidden-row trap. Runtime DOM proof must show the component body/content rectangle uses the available slot height after padding.
- Do not render duplicate refresh, download/export, copy/share, period/date, global-filter, or toolbar controls in business widgets when the selected template already owns those controls. Duplicate visible controls are `VIS-DUPLICATE-CONTROL` or `RPT-SHELL-DUPLICATE` unless component ownership is explicitly declared and the corresponding template control is disabled/hidden.
- Do not implement parent-block title/body chrome as a shell-level title band, raw copied HTML wrapper, or uninspectable one-off CSS. Styled parent blocks must declare `blockChromePattern` or an inherited default through widget/composite-owned config, props, scoped styles, or DOM attributes, and must preserve chart/table/list body floors.
- Template `filters[]` is for horizontal constraints; schema-changing perspectives belong in nav/page/route/tab/segment/perspective state.
- Template data sources must declare or inherit configurable `emptyFilterValues`; aggregate/subtotal rows must use a distinct `aggregateValue`, `rowRole`, or typed key and must not use `all` as a business primary key unless an upstream legacy contract is isolated, adapted, and downgraded from ready.
- Standard chart/table widgets must use their declared renderer and data contract. Ordinary ECharts axis charts such as line, bar, scatter, quadrant, and combo default multi-series legend placement to top center when data exists and space permits; single-series or single-metric legends are hidden by default unless an explicit `legendVisible: true` exception is documented. Proportion/pie/donut/rose and sunburst charts default visible category/hierarchy legends on the right; radar charts show right-side legends for multi-series comparison and hide single-series legends by default. Axis charts configure Y-axis units through `yAxis.name`/axis-unit metadata, keep raw Y-axis tick labels unit-free, place Y-axis titles on the left/right side, place X-axis titles at the bottom, use compact `grid.top/right/bottom/left`, put target/reference labels at `insideEndTop`, and compute dynamic Y-axis ranges for NPS/score/rate/current-vs-target charts; tooltip/detail carries exact value plus unit.
- `npm run validate:dashboard` must pass the stack and geometry contracts before readiness: package dependencies, Vue 3 `createApp`, Element Plus `app.use(ElementPlus, ...)`, Element Plus CSS, ECharts runtime proof when chart visualTypes are present, required list row contracts, compact/internal action-list row budget, post-layout axis-chart container floors (`contentW >=300px`, `contentH >=200px`), squeeze-risk strategy declaration for narrow/short/dense blocks, full-axis chart `chartBodyH >=180px`, or explicit `compact-sparkline` exception.
- When a runnable URL exists, `npm run visual:geometry -- --url <url>` failures at `major` or above block `ready`. Do not bypass geometry audit by relying only on manual screenshot confirmation. List-row `scrollHeight > clientHeight + 1`, hard-hidden list overflow, axis chart containers under `300px` wide or `200px` high, squeeze-risk containers without strategy metadata, axis chart body below `180px`, plot-height failures, missing multi-slot titles, summary-area clipping, and component-slot text/value clipping are direct readiness failures. For KPI/core-value widgets, `minor` clipping is not acceptable: visible value clipping, top/bottom cut-off, reflection overlap, or internal text overflow blocks readiness until the component config or component example is changed.
- Do not accept configured interactions without runtime effect evidence. Every drilldown/jump/modal/drawer/popover interaction must be triggered in the running page and verified through URL/state/modal/drawer/popover DOM evidence, not only through `actions` config presence.
- Template validation scripts or config contracts do not replace runtime proof. Widget contracts such as `compositePanelContract`, `analysisInsightContract`, chart/table option contracts, and control ownership must map to DOM/CSS/renderer/browser evidence before readiness.
- Changed copied-template source files require code-ledger read/create evidence and post-change version entries.
- Do not mark a configured report project ready when `DELIVERY_INDEX.md` was not read or initialized before edits, or when it lacks a post-change entry for the current work.
- Do not ship bundled reusable templates with internal development history in `DELIVERY_INDEX.md` or long `__change_logs__` ledgers. Bundled templates keep initialized process records; copied/configured projects keep the user's real project history.
- Do not mark a configured project ready when `npm run ledger:init` was skipped despite a missing baseline or `npm run ledger:check` fails before handoff.
- Do not mark a configured prototype `ready` for technical solution, backend/data-service design, frontend integration, or testing handoff when `docs/prototype-data-summary.md` is missing, generic, stale relative to `dashboard.config.ts`, `dashboard.dataset.json`, `dataSources/registry.ts`, widget data contracts, generated conclusion rules, filters, or interactions, or lacks Metric To Interface And Source Mapping, a Mock API To HTTP API Replacement Matrix, and backend interface method rows with service method names, request/response DTOs, frontend consumers, binding/adapter mapping, permission/cache/error notes, and mock-to-real replacement guidance.
- Do not expect a `change_logs` folder. The required file-level code-ledger path is the same-directory sidecar `__change_logs__/<code-file-name>.changes.md`; absence of `change_logs` is normal, while absence of required `__change_logs__` sidecars for changed code blocks readiness.
- Load `template-routing-and-implementation-gates.md` before selecting, copying, editing, or accepting bundled report templates.
