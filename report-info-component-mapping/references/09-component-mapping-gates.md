# Component Mapping Gates

Detailed mapping constraints moved out of `SKILL.md`. Load this for implementation-ready binding matrices, mock/data contracts, control semantics, or component routing reviews.

## Hard Constraints

- A component is valid only when it answers a named business question.
- A report component is valid only when it participates in a decision path: state, target/baseline, trend, driver, abnormality, detail, trust, or action. Generic KPI cards and decorative charts without this role are rejected.
- A report mapping must declare one primary decision question. A bundle that only lists metrics, charts, or fields without the lead question is `RPT-NO-PRIMARY-QUESTION` and stays `partial`.
- A report mapping must declare one overall conclusion before primary sections/components are finalized. Missing total judgment is `RPT-NO-OVERALL-CONCLUSION` and stays `partial`.
- Every primary section must declare a section conclusion and its role in explaining the overall conclusion. Missing or unrelated section logic is `RPT-ORPHAN-SECTION` and stays `partial`.
- Every primary component must declare `conclusionChain` with `overallConclusionId`, `supportingSectionId`, `sectionConclusionId`, role, and evidence verb. A component that cannot explain a section conclusion is `RPT-ORPHAN-COMPONENT` and must be moved to detail/interaction/handoff or removed.
- Primary metrics must form a relationship network, not a flat list: result metric -> driver metric -> dimension/object -> detail record -> action/source. A flat KPI wall is `RPT-FLAT-METRIC-LIST` unless the page is explicitly a narrow status-monitoring cockpit with a documented scan task.
- KPI dashboards and metric-heavy reports must turn the primary metric relationship network into `metricDrilldownContract`: root metric, result/diagnosis/process/action layers, triggers, payload fields, target route/component, context inheritance, state rules, and validation cases. Missing contract coverage is `RPT-MISSING-METRIC-DRILLDOWN` unless a scoped static exception is documented.
- Every primary metric needs a comparison basis: target, prior period, same period last year, benchmark, historical range, threshold, or denominator/total for shares/rates. Missing comparison is `RPT-MISSING-COMPARISON` and the metric cannot be the primary judgment anchor.
- Decision reports place judgment/evidence before dense detail. Detail-first layout is `RPT-DETAIL-FIRST` unless the report is explicitly detail-query, reconciliation-first, audit-first, or source-record lookup.
- Every primary judgment should carry a next path: tooltip, drilldown, drawer, cross-filter, export, fullscreen, source jump, action route, or explicit presentation-only static exception. Missing next path for an action/diagnosis report is `RPT-NO-SO-WHAT`.
- Visible drilldown-route lists, metric-drill trees, binding matrices, or implementation chains are `RPT-DRILLDOWN-LIST-VISIBLE` unless the user explicitly requested visible documentation. Expose drilldown through user actions, breadcrumbs, drawers, detail tables, jumps, exports, or action routes.
- Visible report content must pass the business-value test: it helps the reader judge, understand evidence/reason, verify trust/source, or take action. Content that does not pass is `RPT-NO-BUSINESS-VALUE`.
- Design-process artifacts are not visible report components by default. 下钻链路清单, 指标清单, component mapping, binding matrix, workflow/gate checklists, dataset field catalogues, pattern-card inventories, and implementation notes shown as result content are `RPT-PROCESS-ARTIFACT-VISIBLE` unless the user explicitly asked for visible documentation or the item is rewritten as conclusion/evidence/trust/action content.
- Every primary metric-bearing component must have formula/denominator, grain, period, source/freshness, numeric display contract, baseline, and owner/action notes when the output is implementation-ready.
- Every visible metric field must declare value type, raw/display unit, display scale, screen precision, tooltip/export precision, rounding mode, null/zero/denominator-zero display, negative-zero handling, small-nonzero behavior when relevant, formula precision policy, and formatter ownership. Ambiguous `0-1` vs `0-100` percent scale or display-only formatted strings are not implementation-ready.
- Generic marketing sections, decorative cards, generic AI/SaaS feature lists, empty slogan panels, and interchangeable icon blocks are not valid report components unless they map to a real user task, data object, decision, evidence, or workflow action.
- Do not let "modern/high-end/tech" copy become an answer atom. Answer atoms must be business meanings such as status, trend, cause, detail, action, evidence, or data trust.
- When a request is phrased as a business viewing intent, such as 看现状, 看目标, 看趋势, 看排名, 看原因, 看行动, 看质量, or 看明细, the mapping must classify `analysisPerspective` before selecting card families or chart types. A component bundle that jumps directly from a vague intent to visual variety without a perspective is incomplete.
- A selected pattern card is valid only when it maps to a visible component, control, data/API requirement, interaction, export/share behavior, operations note, or validation case. Otherwise mark it as a gap or backlog item.
- A screenshot/sample-derived reusable style is valid only when it declares `styleGeneralization` with source role, status, canonical pattern reference, pattern fields, adaptive variables, renderer owner, fallback, and `textOnlyReproduction: true`. Raw image paths, image embeddings, or OCR-only text are not accepted as durable component mapping evidence.
- Metric-card mappings that use reusable KPI patterns must carry executable evidence bindings, not only a visual enum. KPI overview cards require `kpiOverviewCardPattern`, `kpiEvidenceBinding.overview`, one overview topic, `2-5` ordered metrics, lead metric id when the selected pattern has a lead zone, one local control type/scope when a control is visible, target and attainment fields for every visible progress track, one compact evidence visual maximum, exact-value payload/detail route, and validation cases for default/changed-control/empty-target/over-limit-metrics states. Single-indicator KPI grid cards require `kpiSingleIndicatorLayoutMode`, `kpiEvidenceBinding.singleIndicator`, exactly one evidence visual, local control type/scope when a control is visible, target and attainment fields when the footer is visible, exact-value payload, and validation cases for default/changed-control/empty-target states. KPI judgment cards require `kpiJudgmentCardPattern`, `kpiEvidenceBinding.judgment`, `judgmentKind`, status or score field, score range or status dictionary, threshold/band fields when visible, direction semantics, exactly one hero visual, comparison strip fields, footer evidence/source fields, exact-value payload, and validation cases for default/changed-control/empty-threshold/out-of-range states. KPI goal execution cards require `kpiGoalExecutionCardPattern`, `kpiEvidenceBinding.goalExecution`, `goalExecutionKind`, direction semantics, actual/target fields, required attainment/gap/progress/milestone fields, formula/denominator behavior, exactly one execution hero visual, comparison strip fields, deadline/remaining-time fields when visible, footer evidence/source fields, exact-value payload, and validation cases for default/changed-control/empty-target/over-target/overdue/no-milestone states. KPI time-series cards require `kpiTimeSeriesCardPattern`, `kpiEvidenceBinding.timeSeries`, `temporalAnalysisKind`, ordered x/y fields, grain, latest period, direction semantics, baseline/cycle/volatility/forecast fields required by the selected pattern, tooltip payload, footer evidence/source fields, exact-value payload, and validation cases for default/changed-period/missing-period/missing-baseline/forecast-unavailable states. KPI comparison analysis cards require `kpiComparisonAnalysisCardPattern`, `kpiComparisonAnalysisEvidenceMode`, `kpiEvidenceBinding.comparisonAnalysis`, `comparisonAnalysisKind`, comparable subject grain/roles, shared metric definition/unit/grain/filter scope, direction semantics, benchmark or variance fields when required, deterministic sort/visible-limit rules, tooltip payload, footer evidence/source fields, exact-value payload, and validation cases for default/changed-period/missing-baseline/missing-benchmark/no-comparable-subject states. `horizontal-axis-scatter-diagnostic` requires `kpiScatterEvidenceMode`, `kpiEvidenceBinding.scatter`, object grain, x/y fields and units, point density strategy, tooltip payload, and any reference/threshold/quadrant/category/size fields. `horizontal-spatial-map-diagnostic` requires `kpiMapEvidenceMode`, `kpiEvidenceBinding.map`, geography fields, projection/map resource, visualMap/legend fields, missing-geo policy, and tooltip payload. `paired-comparison-diagnostic` requires `kpiComparisonEvidenceMode`, `kpiEvidenceBinding.comparison`, comparable left/right pane fields, shared unit/grain/filter scope, baseline labels, delta/gap/conclusion fields, and validation cases. Missing any of these keeps readiness `partial`.
- Action recommendation mappings require `analysisPerspective: actionRecommendation`, `componentType: card`, `visualType: action-recommendation-card`, controlled `actionRecommendationCardPattern`, controlled `actionEvidenceBodyMode`, `actionRecommendationEvidenceBinding`, one trigger metric, one primary evidence body, `1-3` actionable items, source/freshness, numeric display contracts for visible values, tooltip payload, detail route, action/execution route, state rules, and validation cases. `strategy-impact-action-card` additionally requires expected uplift/cost/period fields when visible; `task-execution-action-card` and `priority-immediate-action-card` require owner/deadline/status/closure fields when execution is visible. Reject or downgrade the card when actions are generic copy, evidence is screenshot-only, impact values are unbound, CTA has no emitted event, or closure ownership is missing.
- Review impact mappings require `analysisPerspective: reviewImpact`, controlled `reviewImpactCardPattern`, controlled `reviewImpactEvidenceMode`, `reviewImpactEvidenceBinding`, event id/name/type, event start/end or marker date, event period, audience/scope when visible, baseline method, before/after windows or control group, visible metric fields, formulas/units/source/freshness, numeric display contracts, tooltip payload, exact-value/detail route, local-control scope and schema impact, renderer owner, fallback, and validation cases. Reject or downgrade the card when event identity is missing, baseline method is missing, "带来/推动/提升" causal wording lacks control-group, difference-in-differences, matched baseline, or documented business-rule proof, exact values are hidden, the evidence body is screenshot-only/decorative, or KPI strips/trend markers/funnel steps/segment bars cannot fit without squeezing.
- Data quality trust mappings require `analysisPerspective: dataQualityTrust`, controlled `dataQualityTrustCardPattern`, `dataQualityEvidenceBinding`, data-quality task, data object scope, dataset/source, period, numerator/denominator policy, formula or rule references, source/freshness/batch metadata, numeric display contracts, tooltip payload, exact-value/detail/source-lineage route, renderer owner, fallback, state rules, and validation cases. Reject or downgrade the card when the scope is vague, the denominator is missing, rule evidence is absent, source freshness/batch is hidden, exact exception rows are unreachable, action/closure ownership is claimed without a route, or the body is screenshot-only/decorative.
- Detail evidence mappings require `analysisPerspective: detailEvidence`, controlled `detailEvidenceCardPattern`, `detailEvidenceBinding`, detail task, source dataset, row grain, primary key, identity fields, default sort or event order, active filters, local-control scope, status/severity dictionary when visible, tooltip payload, exact-value/detail/export route, renderer owner, density limit, fallback, state rules, and validation cases. Reject or downgrade the card when row proof is missing, the body is screenshot-only/decorative, exact rows are unreachable, local controls duplicate shell/global controls, or rows/columns/steps are unbounded without pagination, scroll, drawer, fullscreen, or table fallback.
- Ranking/leaderboard/Top N mappings require `visualType: ranking-list`, controlled `rankingCardPattern`, `rankingEvidenceBinding`, named ranking mode, metric/unit, source-side Top N or bounded-local exception, visible-count budget, object id/name fields, value field, rank field, deterministic sort and tie-break fields, total/share policy when share is visible, numeric display contracts, overflow strategy, tooltip payload, and detail/action path. If row order is only decorative, share denominator is missing, all rows are fetched/paged before client-side global ranking, or Top N has no view-all/table/drawer/scroll fallback, readiness stays `partial`.
- Pareto mappings require `visualType: ranking-list`, controlled `paretoCardPattern`, `paretoEvidenceBinding`, additive value field, total field, contribution share field, cumulative value/share fields, threshold share, boundary item or key-factor flag, descending rank, tie-break fields, long-tail aggregation/fallback policy, numeric display contracts, tooltip payload, and detail/action path. If the metric is non-additive, rows are unsorted, cumulative share is computed only over the visible Top N without disclosure, the 80% or configured threshold boundary is missing, or long-tail handling is undefined, reject Pareto and route to ranking/table/chart fallback.
- Composition/share/structure/market-share mappings require `visualType: composition-card`, controlled `compositionShareCardPattern`, `compositionShareEvidenceBinding`, metric/unit, category id/name, value, total, share, denominator policy, order rule, `Top N + 其他` or small-slice merge policy, numeric display contracts, tooltip payload, and detail/action path. Hierarchy patterns require parent/level fields and parent-vs-total share fields; market-share and concentration patterns require declared market denominator and CR/HHI fields when visible; map structure requires real geography fields. Reject or downgrade the card when the denominator is missing, shares do not reconcile to 100% within tolerance, negative values feed area/angle/width/bubble size, `其他` is visible without policy, all-zero data is drawn as fake slices, or exact values are hidden with no tooltip/detail route.
- Decomposition/attribution/contribution/hierarchy mappings require `visualType: decomposition-card`, controlled `decompositionAttributionCardPattern`, `decompositionAttributionEvidenceBinding`, root metric/unit/value fields, pattern-specific formula/node/link/stage/time/matrix fields, `signRule`, `reconciliationPolicy`, residual/other policy when the child values do not exactly close to the root, numeric display contracts, tooltip payload, and detail/action path. Attribution wording additionally requires `attributionMethod` or a named diagnostic model; otherwise downgrade to contribution/driver wording. Reject or downgrade the card when formula operators are missing, signed values are rendered as unsigned share/ring evidence, Sankey/path links lack source-target-value fields, hierarchy rows have orphan nodes or unstable IDs, dense matrices have no row/column bound, residual/other is hidden, root/child values do not reconcile under the declared policy, or exact values are hidden with no tooltip/detail route.
- Distribution/interval/density/quantile/boxplot mappings require `visualType: distribution-card`, controlled `distributionAnalysisCardPattern`, `distributionAnalysisEvidenceBinding`, sample grain/count, metric/unit, value or bucket fields, selected bucket/density/quantile/boxplot/heatmap/geography/comparison fields, interval policy, density policy when smoothing is shown, missing/zero/out-of-range policy, numeric display contracts, tooltip payload, detail/action path, and responsive fallback. Reject or downgrade the card when density lacks sample count and estimator/bandwidth/precomputed-density proof, boxplot lacks Q1/median/Q3/whisker/sample count, intervals lack lower/upper bounds or sort order, heatmap cells do not distinguish missing from zero, geographic cards lack map/geo fields, exact bucket/statistic audit is hidden, or visible cells/buckets/points exceed the density budget without aggregation/detail route.
- Time-pattern mappings require `analysisPerspective: timePattern`, controlled `timePatternCardPattern`, `timePatternEvidenceBinding`, ordered time grain, period start/end, metric/unit, aggregation, numeric display contract, missing-vs-zero policy, tooltip payload, exact-value route, local-control scope, density limit, validation cases, and responsive fallback. Calendar cards require date/month/weekday/status fields; schedule lane cards require event start/end/lane/status fields; time-slot and comparison cards require slot order and series fields; heatmap cards require row/column time fields and sample count; peak-valley cards require peak, valley, spread, and selected-period fields; forecast warning cards require forecast horizon/value, threshold/status, and uncertainty or warning fields. Reject or downgrade the card when the question is only a generic trend, calendar grain is missing, controls have no data effect, missing and zero look the same, peak/valley is not computable, forecast warning lacks threshold/status evidence, or exact time-cell values are hidden.
- Anomaly/risk/warning/outlier mappings require `visualType: anomaly-card`, controlled `anomalyAnalysisCardPattern`, `anomalyAnalysisEvidenceBinding`, `analysisPerspective: anomalyRisk`, one diagnostic lens, severity/status dictionary, current/baseline or threshold evidence, occurrence/update/freshness time, impact or affected-object fields when visible, numeric display contracts, tooltip payload, detail/action path, validation cases, and responsive fallback. Reject or downgrade the card when severity/status is missing, the diagnostic lens is only decorative, baseline/threshold/time proof is absent, impact/object/action context is absent, or map/relation/scatter/matrix/gauge/list/table evidence lacks its required data shape.
- Matrix decision mappings require `visualType: matrix-decision-card`, controlled `matrixDecisionCardPattern`, `matrixDecisionEvidenceBinding`, `analysisPerspective: matrixDecision`, one decision task, object grain, required axes/dimensions/priority/layer fields for the selected pattern, deterministic classification rule, threshold or band rule when visible, density and label rules, tooltip payload, exact-value route, renderer owner, fallback, and validation cases. Reject or downgrade the card when axes/dimensions/priority/layer fields are missing, classification is decorative or manual-only, threshold/rule evidence is absent, dense matrices/points/lanes/trees have no aggregation/scroll/fullscreen/table fallback, missing cells and zero cells are conflated, or exact object/cell evidence is hidden.
- Market movement mappings require `visualType: market-analysis-card`, controlled `marketAnalysisCardPattern`, `marketAnalysisEvidenceBinding`, `analysisPerspective: marketMovement`, one market task, instrument identity when instrument-level, quote/OHLC/breadth/volatility/risk-return fields required by the selected pattern, declared market rise/fall color convention, source/freshness, numeric display contracts, tooltip payload, exact-value route, renderer owner, fallback, and validation cases. Reject or downgrade the card when symbol/instrument identity is missing, quote/change/time/source fields are missing, K-line lacks valid ordered OHLC, breadth totals lack denominator reconciliation, volatility lacks formula/window/threshold basis, colors do not declare market convention, delayed/closed/halted/stale states are hidden, dense candles/watchlist rows/heatmap cells have no zoom/scroll/TopN/fullscreen/table fallback, or exact market values are hidden.
- Relationship/correlation/association/influence mappings require controlled `relationshipAnalysisCardPattern`, `relationshipAnalysisEvidenceBinding`, `analysisPerspective: relationshipInfluence`, relation task, entity grain, source/target or pair/node/edge fields, metric/strength/direction/method fields, density and label rules, tooltip payload, exact-value route, renderer owner, fallback, and validation cases. Reject or downgrade the card when pair fields, node-edge schema, source-target-value links, parent-child relation, two numeric axes, or ordered time rows are missing for the selected pattern; when influence wording lacks method/model/business-rule proof; when dense networks/matrices/points/links lack aggregation/search/scroll/fullscreen/table fallback; or when exact relationship evidence is hidden.
- Spatial/map/geographic analysis mappings require `visualType: spatial-map-card`, controlled `spatialAnalysisCardPattern`, `spatialAnalysisEvidenceBinding`, `analysisPerspective: spatialDistribution`, geography grain, region code or longitude/latitude/origin-destination fields, map resource/projection, primary metric/unit/period, visualMap/legend rule, label rule, missing-geo policy, density limit, numeric display contracts, tooltip payload, exact-value route, renderer owner, state coverage, and responsive fallback. Reject or downgrade the card when geography is only an organization label, layer colors/sizes/flows/radii are not tied to named fields, the map can only be understood from a screenshot, the legend or controls would cover important geography, exact spatial values are hidden, or dense points/flows lack clustering, heatmap, Top N, fullscreen, table, or drawer fallback.
- Multidimensional feature/profile/tag mappings require controlled `multiDimensionalFeatureCardPattern`, `multiDimensionalFeatureEvidenceBinding`, `analysisPerspective: multiDimensionalProfile`, one feature task, object/dimension/tag fields required by the selected pattern, scale policy for radar or parallel profiles, density and label rules, tooltip payload, exact-value route, renderer owner, fallback, and validation cases. Reject or downgrade the card when object identity is decorative, radar dimensions lack shared or standardized scale, tag colors/weights/statuses are not tied to fields, tag rules/source/freshness are absent for tag-detail tasks, exact feature/tag values are hidden, or dense dimensions/tags/subjects lack Top N, drawer, fullscreen, table, or simpler bar/list fallback.
- For sample/source restoration, do not promote a visible sample module to `must-have` only because it exists in the source. Mark it as `businessRequired` only when it directly answers the user's stated report question; otherwise use `sampleStructure` or `optionalEnhancement`.
- For status-overview reports, process/path/flow diagrams are secondary unless the core question explicitly asks for value chain, dependency, lineage, transmission, process conversion, or flow attribution.
- A component without a data source is decorative unless it is explicitly static narrative.
- Analysis & Insight components must be mapped as decision-support components, not loose copy. When `componentType` is `text-summary`, `card`, `task`, or `custom` and the content is a conclusion, insight, anomaly/risk explanation, attribution, recommendation, definition, data-quality note, forecast, annotation, explanatory empty state, or permission/no-result/delay note, declare `analysisInsightContract` with subtype, family, conclusion, evidence, affected object, comparison/change value when relevant, action/detail path when relevant, trust/source/freshness/definition when relevant, local-filter scope, tooltip payload, and validation rules.
- Conclusion/insight/summary/interpretation mappings require `analysisPerspective: conclusionInsight`, `visualType: text-summary`, `analysisInsightContract.subtype: conclusion-card`, controlled `conclusionCardPattern`, controlled `conclusionEvidenceBodyMode`, `conclusionEvidenceBinding`, one main conclusion, one primary evidence body, source/freshness, numeric display contracts for visible values, tooltip/detail route, and validation cases. Reject or downgrade the card when the body is only an icon/accent-color variation, when KPI/trend/structure/formula/segment/action evidence lacks required fields, when the card can only be understood from a screenshot, or when the conclusion is generic and not traceable to the visible evidence.
- Definition/help mappings require `analysisPerspective: definitionHelp`, `visualType: text-summary`, controlled `definitionHelpCardPattern`, `definitionHelpEvidenceBinding`, metric id/name, definition text or dictionary route, formula or explicit non-formula reason, scope/include/exclude/source/freshness fields when relevant, numeric display contracts, tooltip payload, exact-value/detail/dictionary route, responsive fallback, and validation cases. Reject or downgrade the card when comparison/trend/share/filter/example bodies are decorative, when denominator/formula/source are hidden, or when the card can only be understood from a screenshot.
- Action recommendation cards are valid only when they package a next-step decision loop: trigger metric -> evidence body -> recommended action -> expected effect or execution owner -> detail/action route. Component mappings must declare `actionRecommendationCardPattern`, `actionEvidenceBodyMode`, `actionRecommendationEvidenceBinding`, trigger metric/formula/grain/period/source, evidence fields, bounded action rows, priority/impact semantics, CTA event/payload, exact-value route, owner/deadline/status when execution is shown, state coverage, and fallback. Use KPI plus operational list when execution rows matter more than analytical evidence; use conclusion cards when the output is only an insight with no executable route.
- Review impact cards are valid only when they package an event review loop: event/action/activity identity -> baseline method -> impact evidence -> result statement -> exact-value/detail route. Component mappings must declare `reviewImpactCardPattern`, `reviewImpactEvidenceMode`, `reviewImpactEvidenceBinding`, event id/name/date/period, before/after or control fields, metric formulas/grain/period/source, state coverage, and fallback. Use trend/comparison/conclusion cards when no event identity exists; use descriptive wording when the method proves correlation or post-period change but not causality.
- Data quality trust cards are valid only when they package a data-confidence question plus audit evidence: quality/completeness/accuracy/consistency/timeliness/anomaly metric -> denominator/rule/source proof -> trend/composition/field/rule/sample evidence -> exact-value/source-lineage/detail/action route. Component mappings must declare `dataQualityTrustCardPattern`, `dataQualityEvidenceBinding`, data object scope, numerator/denominator policy, formula/rule references, source/freshness/batch, numeric display contracts, state coverage, and fallback. Use ordinary KPI, anomaly, composition, table, or definition cards when the surface does not own data-quality trust context.
- Detail evidence cards are valid only when they package row-level proof or drilldown context: row grain -> identity/default order -> bounded detail evidence -> exact-value/detail/export route -> states. Component mappings must declare `detailEvidenceCardPattern`, `detailEvidenceBinding`, primary key, identity fields, local-control scope, renderer owner, density limits, state coverage, and fallback. Use ordinary Detail Table, operational-list, chart, or drawer patterns when the surface does not own detail-evidence context.
- Do not map a generic "智能洞察" or "分析说明" card without concrete evidence, affected object, next action, or data-trust context. If the source data cannot support the conclusion, mark the insight as `insufficient` and route to an explanatory state instead of inventing copy.
- First-screen cards and large panels cannot remain unbound "暂无数据" placeholders.
- Mock data must make KPI totals, chart totals, table rows, drawers, and exports reconcile under the same active filters.
- Mock/offline data must not be unrealistically clean when it is used as prototype evidence. Include realistic variance, missing/zero/extreme values, non-default filter changes, long labels/high-cardinality dimensions, freshness/source metadata, and permission-limited cases where relevant.
- A report page with only overview charts and no detail table, drawer, drilldown, export, owner, or action path cannot be marked implementation-ready unless the report is explicitly static/read-only and that limitation is documented.
- Multi-period filters, trends, MoM, YoY, quarter, or rolling-period views require complete matching time rows.
- Data completeness must be checked before filter binding. Do not finalize `filterMap`, `filterFields`, `requiredFilters`, API params, or resolver params until the underlying option data, row grain, required fields, default/non-default states, and resolver/API support are present or explicitly marked as gaps.
- A filter without affected components is dead UI unless it controls navigation or permissions.
- Control semantics must be classified before binding: `perspective-switch`, `global-filter`, `local-filter`, or `drilldown-param`. A control that changes the metric set, component semantic role, report subject, business-domain vocabulary, table columns/header groups, metric口径, or component collection must not be placed in ordinary/global/local filter surfaces.
- Component-internal `local-filter` is allowed only when it affects the current component or explicitly declared local group, preserves the component schema, and operates on an already fetched/bounded component dataset. It must declare the affected component IDs, local dataset/grain, field mapping, default value, non-default state, and whether the filter is display-only or row-scope local.
- If a supposed local filter changes metric names, formulas/口径, table columns, component collection, first-level report perspective, permission scope, pagination, backend aggregation, export scope, or another component's data, classify it as `perspective-switch` or `global-filter` instead of `local-filter`.
- Business domain, report theme, management object, subject area, or first-level perspective belongs in navigation, tab, segment, route, or perspective layer. It may update active filters internally, but it must not be represented as a normal filter control without documenting the schema impact.
- Data-bearing filter options must derive from dimension data, fact data, or a resolver unless they are stable enums.
- Filter option `meta` may contain only dimensional attributes such as label/name aliases, sort order, permission scope, description, icon, stable category tags, or disabled reason. Dynamic metrics, KPI values, percentages, rankings, and status lights must come from business fact datasets or resolvers, not option `meta`.
- Percentage, ranking, and status-light values shown in perspective navigation must declare metric lineage: `sourceDataset`, `field/formula`, `grain`, `affectedFilters`, and `periodBehavior`. Do not place business/period-sensitive KPIs in `filterData.meta` unless they are explicitly labeled as static display copy.
- Filter IDs must map explicitly to dataset fields or query params.
- A primary/global filter expected to affect a component must map through `filterMap`, `filterFields`, `requiredFilters`, query params, or resolver params. Do not list that filter in `ignoredFilters`; `ignoredFilters` is only for components whose business scope is intentionally invariant under that filter and must be labeled in the scope notes.
- Mock/offline data must have the same grain as the affected primary filters. View, month, period, organization, industry, status, or scenario switches need distinct rows or a resolver that returns different component values; one default snapshot plus selected-state changes is not an implementation-ready contract.
- Filter/query execution must distinguish global/page-level filters from component-internal filters. Global/page-level and permission filters should execute through SQL `WHERE`, source/provider/repository queries, resolver params, Redis/precompute keys, or equivalent source-side scope. Component-internal filters may run locally on the already fetched component dataset. Do not design pages that fetch/build all candidate data and then apply global filters, pagination, ranking, grouping, or aggregation locally.
- Clickable visual elements must define emitted event, target action, parameters, permission behavior, and stale-state behavior.
- Drawers, exports, jumps, refresh, and fullscreen views must reuse the same filtered context as the source component.
- Funnel charts are valid only for ordered conversion/process questions with a shared population or documented cohort logic. Component mappings must declare stage dataset/schema, stage order, metric basis/unit, entry/final values, entry share, stage conversion, drop value/rate, total conversion, stage density, target/comparison behavior when present, label strategy, component-local filter semantics, tooltip/detail payload, and table/path/Sankey/bar fallback for unordered stages, ranking/comparison-only tasks, time trends, multi-branch paths, mixed口径 stages, or too many stages.
- Waterfall charts require additive contribution logic.
- Matrix/time/calendar/correlation heatmaps are valid only for two-dimensional intensity, pattern, utilization, cohort, risk, or correlation reading. Component mappings must declare row dimension, column dimension, value metric, aggregation grain, unit, color-scale/visualMap rule, missing-vs-zero behavior, row/column density, label sampling, value-label threshold, tooltip payload, and table/detail fallback for exact cells or high-cardinality matrices.
- Parallel-coordinate charts are valid only for `3+` metric object profile, similarity, anomaly, Top/selected comparison, or multi-factor screening tasks. Component mappings must declare object/sample dataset/schema, `3-12` dimension fields, meaningful dimension order, per-axis unit/range/direction, independent or standardized scaling, sample count/density, line opacity/sampling/aggregation strategy, group/highlight semantics, brush behavior when present, tooltip/detail payload, and table/scatter/bar fallback for `<3` dimensions, `>12` dimensions, too many samples, or exact-audit tasks.
- Path/user/process path charts are valid only for ordered movement from start to end, such as behavior journey, conversion path, workflow/approval flow, task handoff, drop-off path, or abnormal path. Component mappings must declare step/node dataset/schema, directed transition dataset/schema, start node, end node, order/layer, metric basis, conversion/drop-off formulas, path depth, Top N and "other" aggregation rule, node/link density, main-path/branch strategy, label strategy, legend/filter separation, interactions, and table/detail fallback for exact node/link audit.
- Funnel charts are valid only for ordered stage retention, conversion, or loss reading. Component mappings must declare stage dataset/schema, order, value, unit, shared population/cohort rule, entry/final values, entry share, stage conversion, drop value/rate, total conversion, stage-count fallback, target/comparison semantics when present, legend/filter separation, interactions, and exact stage tooltip/detail; if the question is only ranking, trend, composition, or complex branching, route to bar/line/pie/path/Sankey/table instead.
- Sankey diagrams are valid only for source-to-target flow, allocation, transfer, conversion, loss, or many-stage distribution tasks.
