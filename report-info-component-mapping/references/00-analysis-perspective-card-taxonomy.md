# Analysis Perspective Card Taxonomy

Use this reference before choosing charts or card styles. It converts a vague business request into a stable analytical perspective, then into answer atoms, card families, and candidate components.

The goal is not to force every report into one perspective. A good report usually has one primary perspective and `1-3` secondary perspectives that form a decision path, such as `currentStatus -> targetProgress -> trendMovement -> causeDiagnosis -> actionRecommendation`.

## How To Use

1. Identify the user's verb: 看现状, 看目标, 看趋势, 看排名, 看原因, 看行动, and so on.
2. Select one `analysisPerspective` as primary. Add secondary perspectives only when they answer a different decision question.
3. Map each perspective to answer atoms and component bundles. Do not jump directly from the perspective to a decorative chart.
4. If a perspective switch is exposed in the UI, classify it as `controlSemantics: perspective-switch`, not an ordinary filter, because it may change metric names, component set, table headers, dimensions, or business vocabulary.
5. Keep exact-value, detail, source, and action paths near the perspective that needs them.

## Composition Rules

- `currentStatus` without `targetProgress`, `trendMovement`, `detailEvidence`, or `actionRecommendation` often becomes a shallow dashboard. Add one evidence path when the surface is decision-facing.
- `targetProgress` should carry target formula, denominator, time progress, missing-target state, and `kpiGoalExecutionCardPattern` when the answer is a card-level target attainment, gap, progress, or milestone execution judgment before visual styling.
- `trendMovement` and `reviewImpact` must declare time grain, baseline/event date, and missing-period behavior. Use `kpiTimeSeriesCardPattern` when the answer is a compact card-level trend, change, YoY/MoM, cycle, volatility, or forecast judgment; split to a full chart/table when the time-series evidence needs dense audit. For `reviewImpact`, declare `reviewImpactCardPattern`, `reviewImpactEvidenceMode`, and `reviewImpactEvidenceBinding` when the card packages event identity, baseline method, before/after KPI strip, event marker trend, funnel lift, segment lift, or summary conclusion. Do not claim causal uplift unless control-group, difference-in-differences, matched baseline, or a documented business rule is present.
- `comparisonDifference` must declare the comparable subject grain, role labels, shared metric/unit/grain/filter scope, direction semantics, deterministic sort, and exact-value tooltip/detail. Use `kpiComparisonAnalysisCardPattern` when the answer is a compact card-level direct comparison, group/segment comparison, competitor position, benchmark distance, or variance/gap diagnosis; split to full chart/table when the comparison requires dense audit.
- `rankingContribution` must declare `rankingCardPattern`, `rankingEvidenceBinding`, deterministic sort, Top N/`其他` rules, visible-row/slot budget, tie-break fields, and exact-value tooltip/detail. When it becomes Pareto, also declare `paretoCardPattern`, `paretoEvidenceBinding`, additive value, cumulative share, threshold boundary, and long-tail policy.
- `compositionShare` must declare `compositionShareCardPattern`, `compositionShareEvidenceBinding`, denominator policy, `Top N + 其他` or small-slice merge rules, share formula, color/category identity, exact-value tooltip/detail, and fallback when the denominator or share reconciliation is invalid.
- `decompositionDriver` must declare `decompositionAttributionCardPattern`, `decompositionAttributionEvidenceBinding`, root metric, formula/node/link/stage/matrix fields as required by the selected pattern, sign rule, reconciliation policy, residual/other policy, exact-value tooltip/detail, and fallback before formula, tree, Sankey, waterfall, or matrix evidence is squeezed. Do not claim attribution or causality without an attribution method or diagnostic model.
- `distributionSpread` must declare `distributionAnalysisCardPattern`, `distributionAnalysisEvidenceBinding`, sample grain/count, metric/unit, selected bucket/density/quantile/boxplot/heatmap/geography fields, interval/density policy, missing/zero/out-of-range policy, exact-value tooltip/detail, and fallback before histogram, density, boxplot, heatmap, or interval evidence is squeezed. Do not claim density, quartiles, or outliers without statistical fields or a named estimator/rule.
- `anomalyRisk` must declare `anomalyAnalysisCardPattern`, `anomalyAnalysisEvidenceBinding`, one diagnostic lens, severity/status dictionary, baseline or threshold evidence, occurrence/update/freshness time, affected object or impact fields when visible, exact-value/detail/action route, and fallback before trend, donut, matrix, map, relation, scatter, gauge, timeline, list, or table evidence is squeezed. Use `visualType: anomaly-card` when the card owns abnormality context; use stricter chart/list/table/KPI patterns only for bare evidence bodies.
- `causeDiagnosis` is not valid without either additive contribution, driver evidence, comparison deltas, or a diagnostic table.
- `processBottleneck`, `conversionRetention`, and `flowTransfer` require ordered stages, directed links, node schema, shared cohort, retention window, conversion/loss formula, or bottleneck fields. For `processBottleneck`, declare `processAnalysisCardPattern`, stage/node/status/timing/bottleneck fields, order/threshold rules, exact-value route, and fallback. For `conversionRetention`, declare `conversionRetentionCardPattern`, `conversionRetentionEvidenceBinding`, conversion-retention task, stage/cohort/window/loss/path/comparison fields, numerator/denominator/formula rules, exact-value route, and fallback. Use bar/table instead when the data is only category comparison.
- `relationshipInfluence` must declare `relationshipAnalysisCardPattern`, `relationshipAnalysisEvidenceBinding`, the relation task, method wording, source/target or pair/node/edge fields, strength/direction/threshold rules when visible, density limits, exact-value tooltip/detail, and fallback. Do not claim influence or causality when the evidence is only correlation, co-occurrence, or similarity.
- `multiDimensionalProfile` must declare `multiDimensionalFeatureCardPattern`, `multiDimensionalFeatureEvidenceBinding`, object/dimension/tag fields, comparable scale policy for radar or parallel profiles, density/label rules, exact-value tooltip/detail, and fallback before profile, radar, comparison, tag cloud, or tag-detail evidence is squeezed. Use `object-profile-summary-card` for first-read object identity, `radar-feature-profile-card` or `radar-feature-compare-card` for shared-scale score dimensions, `dimension-score-breakdown-card` when exact dimension ranking matters, `feature-comparison-matrix-card` or `feature-bubble-comparison-card` for multi-subject comparisons, and tag patterns for taxonomy/status/cloud/detail tasks. Do not plot mixed raw units on one radar without standardization and raw-value disclosure.
- `populationObject` must declare `populationObjectCardPattern`, `populationObjectEvidenceBinding`, object/group identity or explicit aggregate policy, denominator policy when shares are visible, required fields by task, density/label rules, exact-value tooltip/detail/action route, renderer ownership, and fallback before profile, segmentation, RFM, lifecycle, behavior preference, channel/source, geography, frequency, churn risk, object KPI, relationship, timeline, score, risk, table, or list evidence is squeezed. Use `population-profile-overview-card` for first-read user/customer/persona summary, `population-segment-composition-card` for group structure, `population-rfm-matrix-card` for RFM quadrants, `population-lifecycle-stage-card` for ordered lifecycle, `population-behavior-preference-card` for category preferences, `population-channel-source-card` for source/channel structure, `population-geographic-distribution-card` for geography-bound cohorts, `population-consumption-frequency-card` for frequency bands, `population-churn-risk-card` for loss-risk cohorts, and the `object-*` patterns for object detail surfaces. Use `multiDimensionalProfile` instead when the main task is feature/radar/tag profile rather than people/object grouping or object 360 evidence.
- `matrixDecision` requires explicit axes, dimensions, priority fields, strategy layer fields, or threshold/classification rules before card selection. Declare `matrixDecisionCardPattern` and `matrixDecisionEvidenceBinding` when the card packages quadrant, matrix, priority, or strategy-layer evidence with header, local controls, legend/rule note, exact-value route, and fallback. Use quadrant patterns when two metrics place objects; matrix patterns when row/column dimensions classify intensity, delta, rating, support, magnitude, or trend; priority patterns when P0/P1/P2/P3 or an ordered urgency/impact/severity model drives action; strategy-layer patterns when L1/L2/L3, goals, actions, dependencies, or execution layers are the primary scaffold. Use relationship, distribution, ranking, comparison, process, or detail perspectives instead when there is no deterministic decision classification. A matrix decision card must include a table/detail fallback for exact object or cell values.
- `marketMovement` requires a market task before card selection: instrument quote, K-line/OHLC reading, market breadth, watchlist/ranking, volatility risk, or risk-return opportunity. Declare `marketAnalysisCardPattern` and `marketAnalysisEvidenceBinding` when the card packages quote identity, local period controls, evidence chart/list, supporting facts, source/freshness, color convention, exact-value route, and fallback. Use quote patterns for current price and change, K-line patterns only with valid ordered OHLC, breadth patterns for up/down/flat or market structure, volatility patterns only with formula/window/threshold fields, watchlist/ranking patterns for bounded row comparison, and risk-return patterns when x/y/size fields share one observation grain. Use `trendMovement`, `distributionSpread`, `rankingContribution`, `anomalyRisk`, or a bare chart/table instead when the component lacks market-specific identity, status, color convention, or freshness.
- `spatialDistribution` requires real geography fields. Declare `spatialAnalysisCardPattern` when the component packages a map with KPI rails, Top lists, legends, controls, coverage, trend, or migration evidence. Do not use a map when region is only an organization label; route to ranking/comparison/table instead.
  - Use `spatial-overview-map-card` for first-read regional status, `spatial-heat-distribution-card` for hotspots, `spatial-comparison-map-card` for baseline/target/level contrast, `spatial-point-distribution-card` for site/object locations, `spatial-trend-change-map-card` for where change happened, `spatial-ranking-map-card` for Top regions with exact ranking, `spatial-coverage-radius-card` for service reach, and `spatial-flow-migration-card` for origin-destination movement.
  - If geography is only supporting evidence inside one KPI card, use `kpiCardPattern: horizontal-spatial-map-diagnostic`. If the map is a standalone chart body without card packaging, use `specializedChartCardPattern: choropleth-ranking-map-card` or the real map/heat/flow chart pattern.
- `timePattern` requires ordered time fields and a named grain. Declare `timePatternCardPattern` when the card packages calendar rhythm, schedule occupancy, period cycle, time-slot distribution, weekday-hour heat, cumulative progress, period comparison, peak-valley diagnosis, or forecast warning. Do not use this perspective for an ordinary single trend line unless the question is specifically about time rhythm, seasonality, time slots, cycle state, or peak/valley behavior.
  - Use `calendar-rhythm-overview-card` for date activity/status, `calendar-schedule-lane-card` for cross-date event/project occupancy, `period-cycle-summary-card` for day/week/month/quarter/year cycle status, `period-progress-status-card` for completion/progress inside a period, `time-slot-trend-card` for intraday high/low movement, `time-slot-share-card` for time-band composition, `weekday-hour-heatmap-card` for weekday-hour intensity, `cumulative-time-curve-card` for running totals, `period-comparison-overlay-card` for this period vs previous periods on the same grain, `peak-valley-diagnostic-card` for peak/valley/spread, and `peak-valley-warning-forecast-card` for threshold forecast risk.
  - Calendar and heatmap cards must distinguish missing from zero. Peak-valley cards must declare peak, valley, spread, and selected period fields. Forecast warning cards must declare forecast horizon, threshold/status, and uncertainty or status evidence.
- `conclusionInsight` packages the answer to "这组数据说明了什么" and must stay evidence-backed. Declare `analysisInsightContract.subtype: conclusion-card`, `conclusionCardPattern`, `conclusionEvidenceBodyMode`, and `conclusionEvidenceBinding` before visual styling. Use `kpi-strip-sparkline` for first-read metric summaries, `trend-compare-chart` for named baseline movement, `composition-structure` for share/cohort/product/channel mix, `formula-driver-chain` for metric drivers or equations, `segment-action-table` for differentiated group actions, and `findings-action-list` when adjacent components already carry the evidence. Do not map a generic "智能洞察" card without visible metric, baseline, source/freshness, exact-value/detail route, and insufficient-data state.
- `actionRecommendation` packages the answer to "接下来该做什么" and must stay trigger-evidence-action backed. Declare `actionRecommendationCardPattern`, `actionEvidenceBodyMode`, and `actionRecommendationEvidenceBinding` when one card owns a trigger metric, evidence chart/list, recommended actions, expected impact, owner/deadline/status, or execution route. Use `diagnostic-evidence-action-card` for metric-triggered recommendations, `strategy-impact-action-card` when expected lift/benefit/cost/period is visible, `task-execution-action-card` when checklist/deadline/status closure is visible, and `priority-immediate-action-card` when an immediate CTA is justified. If the surface is only a plain to-do list without metric trigger/evidence, route to `operational-list` with `progress-task-list` instead.
- `definitionHelp`, `dataQualityTrust`, and `detailEvidence` are trust/evidence perspectives. They can be secondary but should not be removed from implementation-ready analytical pages when metric口径, source, row proof, quality rule proof, or export matter. For `definitionHelp`, declare `definitionHelpCardPattern` and `definitionHelpEvidenceBinding` when the component packages metric meaning, formula, scope, source, denominator, conditions, comparison, trend, share, or calculation examples. For `dataQualityTrust`, declare `dataQualityTrustCardPattern` and `dataQualityEvidenceBinding` when the component packages quality score, completeness, accuracy, consistency, timeliness, uniqueness, exception samples, source freshness, reconciliation, or action closure; the binding must include data object scope, denominator policy, formula or rule references, source/freshness/batch metadata, exact-value/detail/source-lineage route, density/fallback rules, and validation cases. For `detailEvidence`, declare `detailEvidenceCardPattern` and `detailEvidenceBinding` when the component packages row proof, record lookup, metric drilldown, composition drilldown, hierarchy drilldown, sample identity/process/result, related records, log audit, access trace, security audit, or task execution trace; the binding must include row grain, primary key, identity fields, default sort or event order, status/severity dictionary when visible, exact-value/detail/export route, renderer owner, density/fallback rules, and validation cases.
- `filterExploration` changes how users explore the report. Treat its controls as `perspective-switch`, `global-filter`, `local-filter`, or `drilldown-param` according to schema impact.

## Contract Notes

For every mapped component, add:

```text
analysisPerspective: controlled value from this file
businessQuestion: concrete question in user language
answerAtom: controlled answer atom
semanticRole: controlled semantic role
componentType / visualType / pattern field
data grain, required fields, formula or source proof
interaction or evidence path
```

When a single component supports multiple perspectives, use one primary `analysisPerspective` and put secondary perspectives in `secondaryAnalysisPerspectives`. Do not duplicate the component only to represent each perspective separately.
